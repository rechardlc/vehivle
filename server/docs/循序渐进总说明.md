# Go 后端循序渐进总说明（一步一步手敲版）

## 1. 这份文档怎么用
- 目标：让你按固定顺序推进，不会“目录很多但不知道先写哪里”。
- 原则：每次只做一个最小可交付切片（接口或业务域），做完再进入下一步。
- 方法：每一步都包含“要碰哪些目录、写什么、完成标准”。

## 2. 建议执行节奏
- 每天 2-4 小时，聚焦 1 个步骤。
- 每天结束前做 3 件事：自测通过、写 5 行复盘、记录明天第一件事。
- 不追求一天写很多，追求每天都能稳定前进。

## 3. 12 步落地路线

### 第 1 步：把服务跑起来（工程壳）
- 目录：`cmd/api`、`internal/bootstrap`、`configs`、`pkg/logger`、`pkg/response`
- 你要做：启动流程、配置加载、日志初始化、健康检查接口、统一响应结构。
- 完成标准：服务能启动，健康检查接口可访问，日志和响应结构稳定。

### 第 2 步：搭 HTTP 基座
- 目录：`internal/transport/http`、`internal/transport/http/router`、`internal/transport/http/middleware`、`internal/transport/http/handler`
- 你要做：路由分组（admin/public）、request_id、recovery、access log 中间件。
- 完成标准：任何接口都能走统一中间件链路并返回统一格式。

### 第 3 步：先立业务语义（Domain）
- 目录：`internal/domain/model`、`internal/domain/enum`、`internal/domain/rule`
- 你要做：角色、车型状态、价格模式等枚举；核心实体；状态流转规则。
- 完成标准：你能不看数据库也解释清楚业务规则。

### 第 4 步：落数据库与迁移
- 目录：`migrations`、`internal/repository/postgres`
- 你要做：先建 3 张主表（admin_users、vehicles、system_settings），再补其余表；建索引。
- 完成标准：迁移可重复执行，基础增删改查正常。

### 第 5 步：做认证闭环
- 目录：`internal/service/auth`、`internal/integration/auth`、`internal/transport/http/handler`
- 你要做：登录、刷新 token、获取当前用户、RBAC。
- 完成标准：未登录拦截、角色权限生效、错误码正确。

### 第 6 步：做分类 + 参数模板
- 目录：`internal/service/category`、`internal/service/param_template`、`internal/repository/postgres`
- 你要做：分类两级结构、排序、启停；模板和参数项维护。
- 完成标准：分类与模板能支撑车型录入，不出现层级混乱。

### 第 7 步：做车型主流程
- 目录：`internal/service/vehicle`、`internal/domain/rule`、`internal/repository/postgres`
- 你要做：创建草稿、编辑、发布、下架、批量状态、预览、复制。
- 完成标准：发布后公开可见，下架后公开不可见，状态流转正确。

### 第 8 步：做媒体与系统设置
- 目录：`internal/service/media`、`internal/integration/object_storage`、`internal/service/system_setting`、`internal/repository/redis`
- 你要做：上传签名、上传回写、全局价格模式、免责声明、联系方式。
- 完成标准：媒体链路可用，价格模式切换后列表和详情口径一致。

### 第 9 步：做公开接口
- 目录：`internal/service/public`、`internal/transport/http/handler`、`internal/integration/cdn`
- 你要做：首页聚合、分类列表、分享校验；发布后必要的缓存刷新策略。
- 完成标准：公开接口只返回已上架内容，下架链接有兜底返回。

### 第 10 步：补审计与错误治理
- 目录：`internal/service/audit`、`pkg/errors`、`pkg/validator`
- 你要做：关键操作审计日志、参数校验统一化、错误码一致化。
- 完成标准：关键操作可追踪，错误返回稳定可读。

### 第 11 步：补测试与接口契约
- 目录：`tests/unit`、`tests/integration`、`tests/e2e`、`api/openapi`
- 你要做：单测覆盖规则，集成测试覆盖数据访问，e2e 覆盖主链路，更新 OpenAPI。
- 完成标准：核心链路自动化可回归，前后端联调有契约。

### 第 12 步：部署与上线准备
- 目录：`deploy/docker`、`deploy/nginx`、`scripts`、`docs/prd-mapping`
- 你要做：容器化、网关配置、上线检查脚本、验收清单逐项打勾。
- 完成标准：测试环境可部署，可观测，可回滚。

## 4. 每个目录做什么（全量速查）

### 根目录层
- `server/`：后端总工作区，按分层组织全部实现。
- `server/cmd/`：可执行程序入口集合。
- `server/cmd/api/`：主 API 进程入口。
- `server/configs/`：环境配置模板与配置加载约定。
- `server/internal/`：后端核心实现，不对外暴露。
- `server/pkg/`：可复用公共组件（无业务语义）。
- `server/api/`：接口契约文档目录。
- `server/migrations/`：数据库迁移脚本与版本管理。
- `server/scripts/`：开发/测试/部署辅助脚本。
- `server/tests/`：测试总目录。
- `server/deploy/`：部署相关配置。
- `server/docs/`：项目内部文档。

### internal 层
- `server/internal/bootstrap/`：初始化和依赖装配。
- `server/internal/transport/`：协议适配层。
- `server/internal/transport/http/`：HTTP 实现总目录。
- `server/internal/transport/http/router/`：路由注册与分组。
- `server/internal/transport/http/middleware/`：中间件链路。
- `server/internal/transport/http/handler/`：请求处理器。
- `server/internal/service/`：业务编排层总目录。
- `server/internal/service/auth/`：认证与会话。
- `server/internal/service/vehicle/`：车型主流程。
- `server/internal/service/category/`：分类管理。
- `server/internal/service/param_template/`：参数模板管理。
- `server/internal/service/media/`：媒体签名与回写。
- `server/internal/service/system_setting/`：全局系统设置。
- `server/internal/service/public/`：小程序公开查询。
- `server/internal/service/audit/`：审计日志编排。
- `server/internal/repository/`：数据访问抽象层。
- `server/internal/repository/postgres/`：PostgreSQL 实现。
- `server/internal/repository/redis/`：Redis 缓存实现。
- `server/internal/domain/`：领域语义总目录。
- `server/internal/domain/model/`：实体和值对象。
- `server/internal/domain/enum/`：枚举常量与合法性。
- `server/internal/domain/rule/`：业务规则与状态机。
- `server/internal/integration/`：外部系统集成。
- `server/internal/integration/auth/`：JWT/密码能力。
- `server/internal/integration/object_storage/`：OSS/COS 对接。
- `server/internal/integration/cdn/`：CDN 刷新策略。

### 其他层
- `server/pkg/response/`：统一响应结构。
- `server/pkg/errors/`：错误码和错误模型。
- `server/pkg/validator/`：参数校验组件。
- `server/pkg/logger/`：日志组件。
- `server/api/openapi/`：OpenAPI 契约文件。
- `server/tests/unit/`：单元测试。
- `server/tests/integration/`：集成测试。
- `server/tests/e2e/`：端到端测试。
- `server/deploy/docker/`：容器化配置。
- `server/deploy/nginx/`：网关配置。
- `server/docs/prd-mapping/`：PRD 与技术映射。
- `server/docs/learning-path/`：学习与实战路线文档。

## 5. 你今天就能开始的第一天任务
1. 只做第 1 步（工程壳），不要跨步。
2. 成功后再做第 2 步（HTTP 基座）。
3. 把你的第一版目录和启动日志截图留存，作为后续迭代基线。

## 6. 防止“学着学着就乱了”的三条纪律
1. 先文档后实现：任何新业务先写到对应目录 README 的“待实现清单”。
2. 先主流程后边界：先跑通登录 -> 发布 -> 公开可见 -> 下架兜底。
3. 先可测后优化：有测试兜底再做缓存、并发、重构。

你只要按这 12 步走，不跳步，2-4 周内就能把这个 V1 后端从 0 推到可企业化交付。

## 一、JWT 链路全景——从登录到鉴权

完成认证闭环后，整条链路贯通：

```
浏览器
  │
  ├─ POST /api/v1/admin/auth/login  { username, password }
  │   └─ Handler.Login
  │       └─ Service.Login
  │           ├─ repo.FindByUsername → bcrypt.CompareHashAndPassword
  │           ├─ jwt.GenerateToken(secret, 2h)   → AccessToken
  │           └─ jwt.GenerateToken(refreshSecret, 7d) → RefreshToken
  │       └─ Set-Cookie: access_token (Path=/api, HttpOnly, SameSite=Lax)
  │          Set-Cookie: refresh_token (Path=/api/v1/admin/auth, HttpOnly, SameSite=Lax)
  │       └─ response { expiresIn: 7200 }
  │
  ├─ GET /api/v1/admin/vehicles（浏览器自动携带 AT Cookie）
  │   └─ middleware.JWTAuth
  │       ├─ c.Cookie("access_token") → jwt.ParseToken
  │       └─ c.Set("current_user_id", "current_username", "current_role")
  │   └─ Handler.List（已有认证上下文）
  │
  ├─ POST /api/v1/admin/auth/refresh（AT 过期时，浏览器自动携带 RT Cookie）
  │   └─ Handler.Refresh
  │       └─ Service.RefreshToken
  │           ├─ jwt.ParseToken(refreshSecret, RT)
  │           └─ jwt.GenerateToken(secret, 2h) → 新 AccessToken
  │       └─ Set-Cookie: access_token=新值
  │
  ├─ GET /api/v1/admin/auth/me（需 AT Cookie）
  │   └─ middleware.JWTAuth → Handler.Me → repo.FindByID → 用户信息
  │
  └─ POST /api/v1/admin/auth/logout
      └─ Handler.Logout → Set-Cookie: MaxAge=0（清除双 Cookie）
```

### 2.3 F5 详解——无感刷新链路为什么断了

这是**前后端联调最隐蔽的断裂点**，值得单独讲清楚。

**前端拦截器的工作前提**：

```
Axios 响应拦截器有两个回调：
  interceptors.response.use(
    (response) => { ... },   // ← HTTP 2xx 走这里（成功）
    (error) => { ... }       // ← HTTP 4xx/5xx 走这里（失败）
  )
```

**之前的问题**：

```
后端 FailAuth → Fail() → c.JSON(http.StatusOK, ...) → HTTP 200
                                                         ↓
Axios 收到 200 → 走 success 回调 → 不进入 error 回调
                                     ↓
requestData 检查 code !== "000000" → throw Error("认证令牌已过期")
                                      ↓
这个 Error 直接到了调用方的 catch → useQuery.onError 跳转登录页
                                     ↓
                              ❌ 无感刷新拦截器永远不触发
```

**修复后**：

```
后端 FailAuth → c.JSON(http.StatusUnauthorized, ...) → HTTP 401
                                                         ↓
Axios 收到 401 → 走 error 回调 → error.response.data 包含业务码
                                   ↓
拦截器检查：code === "A00001" && message.includes("过期")
                                   ↓
          ✅ 自动调 POST /auth/refresh → 签发新 AT → 重发原始请求
```

**类比前端**：就像 Axios 拦截器只会在 `Promise.reject` 时执行 `.catch`，`response.use` 的第二个回调只在 HTTP 非 2xx 时执行。如果后端所有错误都返回 200，你的 error 拦截器形同虚设。

**tech.md 对齐**：

| 错误码 | HTTP 状态码（tech.md 规定） | 之前实现 | 修复后 |
|--------|---------------------------|---------|--------|
| A00001 | **401** | 200 | **401** ✅ |
| A00004 | **403** | 200 | **403** ✅ |

## 三、核心知识点

### 3.1 `json:"-"` —— 为什么 PasswordHash 绝对不能序列化

```go
PasswordHash string `json:"-" gorm:"column:password_hash;not null"`
```

- `json:"-"` 告诉 `encoding/json`：**序列化时跳过此字段**
- 类比前端：相当于 TypeScript 接口中不导出某个字段，确保 `JSON.stringify` 时不泄露
- 如果不加 `-`，`response.Success(c, user)` 会把密码哈希写入 JSON 响应
- 虽然 bcrypt 哈希本身不能反推密码，但泄露哈希仍可被用于离线暴力破解

### 3.2 登录失败为什么要统一错误信息

```go
// ✅ 正确：统一返回
return nil, ErrInvalidCredentials // "用户名或密码错误"

// ❌ 错误：分别返回
return nil, errors.New("用户不存在")  // 攻击者可枚举用户名
return nil, errors.New("密码错误")    // 攻击者确认用户名存在后暴力破解密码
```

- 统一错误信息是企业安全标准（OWASP 认证最佳实践）
- 攻击者无法通过错误信息区分「用户名不存在」和「密码错误」
- 类比前端：前端 toast 只显示「用户名或密码错误」，不要显示更具体的原因

### 3.3 `SameSite=Lax` vs `SameSite=None`

| SameSite 值 | 效果 | 安全性 |
|-------------|------|--------|
| `Strict` | 任何跨站请求都不携带 Cookie | 最严，但影响导航体验 |
| **`Lax`** | 顶级导航 GET 可携带，**跨站 POST 不携带** | **推荐**——管理后台写操作均为 POST |
| `None` | 所有请求均携带（需 `Secure=true`） | 最松，CSRF 攻击有效 |

之前代码用了 `SameSiteNoneMode`，意味着恶意网站可以发起 POST 请求并自动携带 Cookie——CSRF 攻击有效。改为 `Lax` 后，跨站 POST 不携带 Cookie，CSRF 失效。

### 3.4 RT Cookie Path 限制 —— 缩小泄露面

```
AT Cookie: Path=/api                    → 所有 /api 请求携带
RT Cookie: Path=/api/v1/admin/auth      → 仅 auth 路径请求携带
```

RT 比 AT 有效期长（7 天 vs 2 小时），泄露影响更大。限制 Path 后，普通业务请求不携带 RT，即使中间人截获普通请求也拿不到 RT。

### 3.5 无感刷新全流程（前后端联调时序）

```
时序 ─────────────────────────────────────────────────────────────
  浏览器                       Axios 拦截器                   Go 后端
    │                              │                            │
    │── GET /admin/vehicles ──────▶│                            │
    │   (Cookie: access_token=过期)│                            │
    │                              │── 转发请求 ───────────────▶│
    │                              │                            │── JWTAuth: ParseToken
    │                              │                            │   → ErrTokenExpired
    │                              │◀── HTTP 401 ──────────────│   → FailAuth("认证令牌已过期")
    │                              │   { code:"A00001",         │
    │                              │     message:"认证令牌已过期" }
    │                              │                            │
    │                              │ 判断: A00001 + "过期" ✅   │
    │                              │ isRefreshing = true         │
    │                              │                            │
    │                              │── POST /auth/refresh ────▶│
    │                              │   (Cookie: refresh_token)  │── ParseToken(RT)
    │                              │                            │── GenerateToken(新AT)
    │                              │◀── HTTP 200 ──────────────│── Set-Cookie: access_token=新
    │                              │   { expiresIn: 7200 }      │
    │                              │                            │
    │                              │ flushQueue() 通知排队请求   │
    │                              │ isRefreshing = false        │
    │                              │                            │
    │                              │── GET /admin/vehicles ────▶│  ← 重发原始请求
    │                              │   (Cookie: access_token=新)│── JWTAuth: ParseToken ✅
    │                              │◀── HTTP 200 ──────────────│── 正常返回车型列表
    │                              │                            │
    │◀── 收到车型列表 ────────────│                            │
    │   (用户无感知)               │                            │
```

**前端拦截器的核心机制**（`client.ts`）：

```typescript
// 关键 1：只在 HTTP 非 2xx 时进入此回调
http.interceptors.response.use(
  (response) => response,                    // 200 走这里
  async (error: AxiosError) => {             // 401/403/500 走这里
    const data = error.response?.data;       // ← 401 时 Axios 也会解析 JSON body

    // 关键 2：精确匹配「AT 过期」，排除其他认证错误（如令牌无效、缺失）
    const isAuthExpired = data?.code === "A00001" && data.message.includes("过期");

    // 关键 3：防止 refresh 请求自身 401 时死循环
    const isRefreshRequest = originalRequest?.url?.includes("/admin/auth/refresh");

    // 关键 4：并发锁 + 队列，避免多个请求同时触发 refresh
    if (isRefreshing) {
      await enqueueRetry();     // 排队等待
      return http(originalRequest); // refresh 成功后重发
    }
  }
);
```

**并发场景**（3 个请求同时 401）：

```
请求A → 401 → isRefreshing=false → 发起 refresh → isRefreshing=true
请求B → 401 → isRefreshing=true  → 进入队列等待
请求C → 401 → isRefreshing=true  → 进入队列等待
                                     ↓
                              refresh 成功
                                     ↓
                              flushQueue() → 请求B、C 的 Promise resolve
                                     ↓
                              请求A、B、C 各自用新 AT Cookie 重发
```

**关键**：Refresh 端点从 Cookie 读 RT（不是从 request body），前端无需手动传递任何 Token。

### 3.6 `RequireRole` 中间件 —— `map[string]bool` 做 O(1) 查找

```go
func RequireRole(allowedRoles ...string) gin.HandlerFunc {
    roleSet := make(map[string]bool, len(allowedRoles))
    for _, r := range allowedRoles {
        roleSet[r] = true
    }
    return func(c *gin.Context) {
        role, _ := c.Get(CtxKeyRole)
        if !roleSet[role.(string)] {
            response.FailAuthDenied(c, "无权访问该接口")
            c.Abort()
            return
        }
        c.Next()
    }
}
```

- `map[string]bool` 初始化在工厂函数中（闭包），每次请求只做 O(1) 查找，不重复构建
- 类比前端：`new Set(allowedRoles)` + `set.has(role)`
- V1 角色仅 `super_admin` 和 `editor`，大多数接口两者都可用

### 3.7 `configs.Validate()` —— Fail Fast 原则

```go
func (c *Conf) validateJWTRequired() error {
    if len(c.JWT.Secret) < 32 {
        return fmt.Errorf("jwt.secret must be at least 32 characters for HS256 security")
    }
    if c.App.Env == "prod" && !c.JWT.CookieSecure {
        return fmt.Errorf("jwt.cookie_secure must be true in production (HTTPS only)")
    }
    ...
}
```

- HS256 密钥长度 ≥ 32 字符是安全基线（256 bit 熵值）
- 生产环境 `CookieSecure=false` 意味着 HTTP 明文传输 Cookie，必须阻止启动
- 类比前端：`z.string().min(32)` 的 Zod schema 校验

## 二、今日完成

### 1. 启动链路的错误处理意识补齐

`cmd/api/main.go` 中启动失败路径从 `os.Exit(1)` 改成 `panic(err)`。

这次学习的关键点不是“panic 一定比 os.Exit 好”，而是理解二者差异：

- `os.Exit(1)` 会立刻退出进程，不会执行 `defer`。
- `panic(err)` 会沿调用栈展开，触发已注册的 `defer`，并输出堆栈信息。
- 真正企业级服务入口通常会进一步升级为：`http.Server` + signal 监听 + context 超时关闭 + `defer b.Close()`。

当前代码里 `defer b.Close()` 仍是注释状态，说明“优雅关闭”还没有真正落地，只是开始意识到 `os.Exit` 会阻断 defer 这类清理动作。

### 2. Gin 中间件顺序和重复日志问题

`bootstrap.Run()` 中保留了：

```text
gin.Recovery()
logger.RequestID()
logger.AccessLog(...)
```

并注释掉了 `gin.Logger()`。

这背后的学习点是：

- `gin.Recovery()` 用于捕获 panic，避免单个请求把整个进程打崩。
- 项目已经有自定义 `AccessLog`，继续打开 `gin.Logger()` 会造成访问日志重复。
- 中间件顺序会影响链路行为：越早注册的中间件越早进入，也越晚退出。

更准确地说，`AccessLog` 的 `c.Next()` 前后构成“洋葱模型”：

```text
AccessLog before
  -> next middleware / handler
AccessLog after
  -> 记录 status、latency、error
```

所以访问日志应该在 `c.Next()` 之后读取状态码和耗时。

### 3. RequestID 从 Gin context 进入标准 context

`logger.RequestID()` 负责：

- 从请求头读取 `X-Request-ID`。
- 如果没有，则生成 UUID。
- 写回响应头 `X-Request-ID`。
- 放入 `gin.Context`。

`logger.AccessLog()` 再把这个 request id 写入标准库 `context.Context`：

```go
ctx := WithRequestID(c.Request.Context(), requestID)
```

这样做的意义是：业务代码、日志库和标准库生态更容易围绕 `context.Context` 协作，而不是所有地方都依赖 Gin 的 `*gin.Context`。

从前端视角类比：

- `gin.Context` 更像某个框架内部的 request 对象。
- `context.Context` 更像跨库都能传递的 request-scoped metadata。

### 4. slog.Handler 包装器的思路建立

`pkg/logger/logger.go` 新增了 `contextHandler`，包一层原生 `slog.Handler`，在日志真正写出前从 `ctx` 中自动提取字段：

- `request_id`
- `user_id`

这是一个很重要的抽象方向：调用方只需要写

```go
logger.Info(ctx, "request", ...)
```

底层 logger 自动把上下文公共字段补上，避免每个调用点重复传 `request_id`。

今日学到的 Go 设计点：

- `slog.Handler` 是日志输出的底层扩展点。
- 匿名嵌入 `slog.Handler` 可以形成轻量包装。
- `Handle(ctx, record)` 是补充上下文字段的关键入口。
- 公共日志字段适合在 handler 层统一注入，不适合散落在每个业务函数里。

### 5. JWT 用户身份向日志上下文传递

`JWTAuth` 中把 `claims.UserID` 写入了：

- `gin.Context`
- 标准库 `context.Context`

这说明今天开始把“认证信息”接到“可观测日志”上。

设计目标是好的：认证通过后，后续日志应能自动带上 `user_id`，排查问题时就能从请求追到具体操作者。

不过当前仍有一个需要特别注意的细节：`AccessLog` 注册顺序早于 admin 路由上的 `JWTAuth`，它在 `c.Next()` 之前创建了本地变量 `ctx := WithRequestID(c.Request.Context(), requestID)`。如果后续 `JWTAuth` 再替换 `c.Request.Context()`，AccessLog 最后记录日志时仍可能使用旧的 `ctx`，从而拿不到后续写入的 user id。

更稳的做法是：`c.Next()` 之后再从 `c.Request.Context()` 取最新 context，并补 request id，或者在 `AccessLog` 的 after 阶段同时读取 Gin context 中的用户字段。

### 6. 配置加载注释和 Makefile 可读性清理

`configs/conf.go` 中补充了 `mapstructure` 和 `.env/.env.{env}` 读取路径的学习注释。

今日要记住的点：

- `mapstructure` 类似 JSON tag，用于把配置文件字段映射到 Go struct。
- `gotenv.Load()` 的相对路径跟当前执行目录有关。
- 本项目默认在 `server/` 目录执行命令，因此 `.env` 和 `.env.{env}` 可以按当前目录读取。

`Makefile help` 中把部分中文说明改成英文，减少不同终端编码下的显示问题。

## 三、当前风险和待闭环点

### 1. user_id 日志注入还需要实测

当前 `contextHandler` 会从标准 context 中读取 `"current_user_id"`，`JWTAuth` 也会写这个 key。

但因为中间件顺序和 `ctx` 变量创建时机，访问日志最终是否稳定带上 `user_id` 还需要实测。

建议下一步用一个受 JWT 保护的 admin 接口验证日志字段：

```text
request_id: should exist
user_id: should exist after auth
status: should exist
latency_ms: should exist
```

### 2. context key 类型仍偏弱

`logger` 包内部的 request id 使用了自定义 `ctxKey` 类型，这是好习惯。

但 JWT 用户信息使用字符串 key：

```go
context.WithValue(ctx, "current_user_id", claims.UserID)
```

字符串 key 容易和其他包冲突。更稳的方式是定义专用 key 类型，并集中提供：

```text
WithUserID(ctx, id)
GetUserID(ctx)
```

### 3. 优雅关闭仍未落地

`main.go` 里已经意识到 `defer b.Close()`，但当前还没有真正启用，也没有把 `r.Run()` 换成可控的 `http.Server`。

下一步如果要进入企业级服务生命周期，需要补：

- `http.Server`
- `signal.NotifyContext`
- `server.Shutdown(ctx)`
- DB/OSS 资源关闭
- 启动失败和运行中失败的日志区分

### 4. ginErr 只读取 public error

`AccessLog` 当前读取：

```go
c.Errors.ByType(gin.ErrorTypePublic)
```

这只能记录被标记为 public 的 Gin error。普通 handler 如果只是直接 `response.Fail(...)`，不会自动进入 `c.Errors`。

后续如果要稳定记录业务错误，需要统一约定：

- handler 返回错误并由中间件集中处理；或
- response helper 同时写入错误上下文；或
- AccessLog 记录统一响应 code。

## 四、学习结论

今天真正学到的是：日志链路不是“打印几行文本”，而是一条跨层传递上下文的工程链路。

在 Go/Gin 项目里，推荐按这个方向理解：

```text
Gin middleware 负责拿到请求现场
context.Context 负责跨库传递请求元信息
slog.Handler 负责统一补充公共日志字段
AccessLog 负责在请求结束后记录结果
JWTAuth 负责把认证身份注入请求上下文
```

当这条链路闭环后，后续排查线上问题时就可以按 `request_id` 追踪单次请求，也可以按 `user_id` 追踪操作者行为。

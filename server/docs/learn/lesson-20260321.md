### 核心概念

| 概念 | 要点 |
|------|------|
| **up / down** | `up` 应用结构变更；`down` 回滚一步（开发环境慎用）。 |
| **版本号** | `000001`、`000002` 递增；已执行过的 `.sql` 不改，新变更用新文件。 |
| **`schema_migrations`** | 由工具维护，记录当前版本，避免重复执行。 |
| **iofs** | 迁移文件用 `fs.FS` 提供，跨平台不依赖 `file://` URL。 |

### 2.2 Go 接口哲学（核心认知转变）

#### "结构体" vs "结构"（中文语境）

| 说法 | 指什么 | Go 关键字 | 示例 |
|------|--------|-----------|------|
| **结构体** | 有固定字段的数据类型 | `struct` | `type Service struct { vehicles VehicleRepo }` |
| **接口** | 一组方法签名的约定 | `interface` | `type VehicleRepo interface { GetById(...) }` |
| **结构**（口语） | 架构、组织、层次 | 无 | "项目结构""分层结构" |

#### Go 接口 vs TypeScript 接口：定义位置不同

| 维度 | TypeScript / Java | Go |
|------|-------------------|-----|
| 接口定义在哪 | **提供方**（如 `repository/`） | **消费方**（如 `service/vehicle/`） |
| 实现方式 | `class Foo implements IBar` 显式声明 | **隐式**：方法签名匹配即满足（鸭子类型） |
| 核心思想 | "我告诉你我长什么样" | **"我只关心你能做什么"** |

**Go 标准库的经典示例**：`io.Reader` 定义在 `io` 包（使用方），`os.File` 从未写 `implements io.Reader`，但它方法匹配，自动就是 `io.Reader`。

#### 本项目的正确做法

```go
// service/vehicle/service.go —— 消费方定义接口
type VehicleRepo interface {
    GetById(ctx context.Context, id string) (*model.Vehicle, error)
    Update(ctx context.Context, v *model.Vehicle) error
}

// repository/postgres/vehicle_repo.go —— 实现方只管实现方法
// 不需要 import service 包，不需要写 implements
func (v *VehicleRepo) GetById(...) (*model.Vehicle, error) { ... }
func (v *VehicleRepo) Update(...) error { ... }
```

**依赖方向**：service 不 import repository 包，两者只通过 `domain/model` 间接关联。在 bootstrap/router 构造时把 postgres 实现注入给 service。

### 2.3 service.go 的四件套模式

当前 `service/vehicle/service.go` 的代码结构是 Go 后端常见的四件套：

```go
// 1. 接口（声明需要的能力）
type VehicleRepo interface { ... }

// 2. 结构体（持有依赖）
type Service struct { vehicles VehicleRepo }

// 3. 构造函数（注入依赖，返回指针）
func NewService(vehicles VehicleRepo) *Service { ... }

// 4. 业务方法（指针接收者，操作依赖）
func (s *Service) Publish(ctx context.Context, ...) error { ... }
```

| 组件 | 作用 | 前端类比 |
|------|------|----------|
| `interface` | 声明依赖契约 | TypeScript 的 `interface`，但定义在使用方 |
| `struct` | 存储状态与依赖 | 类的实例字段 / React 组件的 state |
| `NewService()` | 工厂函数 | `new Service()` 或 `createService()` |
| `(s *Service) Method()` | 指针接收者方法 | 类的方法 `this.method()` |

## 复盘

- **上午**：「连上库」≠「有表」；`make migrate-up` 后才有真实 schema 与 `schema_migrations`。Windows 下迁移源优先 **iofs**。
- **下午**：全链路诊断发现 4 处断裂；核心原因是 Handler 直接传 `*gorm.DB` 给 Service，而 Service 期望接口类型。
- **认知转变**：Go 接口由消费方定义（`service` 定义 `VehicleRepo`），实现方（`repository/postgres`）隐式满足，不需要 `implements` 关键字。这与 TypeScript/Java 的「接口定义在提供方」截然不同。
- **四件套模式**（interface → struct → NewXxx → method）是 Go 服务层的标准写法，后续所有 service 都可套用。
- `internal/repository/` 目录下不需要单独定义接口文件，接口已归属到各 service 包中。

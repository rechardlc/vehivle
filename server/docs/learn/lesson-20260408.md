## 一、MinIO 客户端封装（infrastructure/oss）

### 1.1 为什么要单独封装

```
infrastructure/oss/client.go   ← 新增
```

- `MinioClient` 结构体持有 `*minio.Client`、`Endpoint`、`PublicURL`、`Bucket`。
- **不直接暴露 `*minio.Client`**，而是聚合访问配置，方便 Handler 拼接公开 URL 等。
- 类比前端：`oss.MinioClient` ≈ 一个带了配置的 axios 实例，Handler 从它取 `Client` 做请求、取 `PublicURL` 拼图片地址。

### 1.2 前端 vs Go 对比

| 概念 | 前端 | Go |
|------|------|-----|
| SDK 客户端 | `new COS({ ... })` / `new S3Client({})` | `minio.New(endpoint, &minio.Options{...})` |
| 配置聚合 | axios 实例 + baseURL | `MinioClient` struct（`Endpoint` + `Bucket` + `Client`） |
| 上传调用 | `cos.putObject(key, file)` | `client.PutObject(ctx, bucket, key, reader, size, opts)` |

## 二、Bootstrap 生命周期扩展

### 2.1 新增 `ossConnPool()` — MinIO 连接初始化

启动顺序变为：**cfg → logger → DB → Ping → OSS → Router**，OSS 为强依赖，启动时连不上即 fail fast。

```go
func (b *Bootstrap) Run() (*gin.Engine, error) {
    // ... gin 初始化、中间件 ...
    b.pgsqlConnPool()  // ① PostgreSQL
    postgres.Ping()    // ② 健康检查
    b.ossConnPool()    // ③ MinIO/S3 ← 新增
    b.injectRouter(r)  // ④ 路由注册
}
```

### 2.2 `normalizeOssEndpoint` — URL 归一化

MinIO SDK 的 `minio.New(endpoint, ...)` 要求 **host:port**（无 scheme）。用户配置可能写 `http://localhost:9000` 或 `localhost:9000`，此函数统一剥离 scheme 并推导 TLS：

| 输入 | 输出 endpoint | 推导 useSSL |
|------|--------------|-------------|
| `http://localhost:9000` | `localhost:9000` | `false` |
| `https://oss.example.com` | `oss.example.com` | `true` |
| `localhost:9000` | `localhost:9000` | 沿用配置值 |

### 2.3 Bucket 自动创建

```go
exists, _ := minioClient.BucketExists(ctx, bucket)
if !exists {
    minioClient.MakeBucket(ctx, bucket, minio.MakeBucketOptions{Region: region})
}
```

开发环境首次启动时自动创建 Bucket，避免手动到 MinIO Console 操作。

### 2.4 错误处理改进：`injectRouter` 返回 error 替代 `os.Exit(1)`

之前 `injectRouter` 内部直接 `os.Exit(1)`，这是**不良实践**：
- 调用方无法捕获错误、无法做清理（如关闭 DB 连接）。
- 测试时 `os.Exit` 会直接终止测试进程。

改后返回 `fmt.Errorf("注册路由失败: %w", err)`，由 `Run()` → `main()` 决定退出策略。

**Go 错误处理原则**：底层函数应返回 error，由最上层（`main`）决定是否退出。类比前端：底层模块抛异常，顶层 `try/catch` 或 `process.exit()` 决定。

## 三、图片上传 Handler

### 3.1 文件结构

```
handler/upload.go   ← 新增
```

`Upload` 结构体持有 `oss.MinioClient` 与 `*gorm.DB`，通过 `NewUpload(ossClient, db)` 构造注入（**2026-04-09** 起上传成功后会 **INSERT `media_assets`**，见 [lesson-20260409.md](./lesson-20260409.md)）。当日初版仅为 `NewUpload(oss)`。

### 3.2 上传流程

```
前端 FormData("file") → Gin FormFile("file") → 类型/大小校验 → 生成对象键 → PutObject → 返回 URL
```

关键设计点：

| 步骤 | 做了什么 | 为什么 |
|------|---------|--------|
| **MIME 白名单** | `allowedImageTypes` map 校验 Content-Type | 防止上传非图片（如 `.exe`） |
| **大小限制** | `header.Size > 10MB` | 避免超大文件占用带宽与存储 |
| **对象键** | `images/20260408/<uuid>.jpg` | 按日期分目录 + UUID，避免文件名碰撞与特殊字符 |
| **`defer file.Close()`** | 函数退出时关闭文件句柄 | Go 没有 GC 自动关闭文件，需手动释放资源 |

### 3.3 `defer` 关键字（前端视角）

```go
file, header, err := c.Request.FormFile("file")
defer file.Close()   // 无论后续是否 return/panic，函数退出前一定执行
```

类比前端：
```javascript
// 类似 try...finally
try {
  const stream = fs.createReadStream(path);
  // ... 使用 stream ...
} finally {
  stream.close();  // ← 这就是 defer 的作用
}
```

### 3.4 `MaxMultipartMemory` 设置

```go
r.MaxMultipartMemory = 10 << 20  // 10MB
```

Gin 默认 `32MB`，限制 multipart 解析时的内存占用。超出部分写入临时文件。`10 << 20` 是位运算：`10 * 2^20 = 10MB`。

## 四、Router 依赖注入扩展

### 4.1 Router 新增 `oss` 字段

```go
type Router struct {
    engine *gin.Engine
    logger logger.Logger
    db     *gorm.DB
    oss    oss.MinioClient   // ← 新增
}
```

`New()` 签名同步扩展为 `New(engine, logger, db, ossClient)`。依赖链：

```
main → Bootstrap.Run() → injectRouter(r)
         ↓                    ↓
    ossConnPool()        router.New(gin, logger, db, ossClient)
         ↓                    ↓
    MinioClient          uploadHandler := NewUpload(r.oss)
                              ↓
                         admin.POST("/upload/images", uploadHandler.UploadImages)
```

### 4.2 健康检查增加 `ossReady`

```go
response.Success(c, gin.H{
    "status":   "ok",
    "message":  "API is running",
    "ossReady": r.oss.Bucket != "",
})
```

健康检查返回 OSS 就绪状态，方便前端/运维判断服务完整性。

## 五、统一响应 `RequestID` 补全

### 5.1 修改前的问题

`Success()`、`Fail()` 等函数构造 `Body` 时，**漏填了 `RequestID` 字段**——结构体定义了但没赋值，JSON 响应中 `requestId` 始终为空字符串。

### 5.2 修改后

所有响应函数（`Success`、`Fail`、`FailNotFound`、`FailMethodNotAllowed`）统一调用 `getRequestID(c)` 填入。

**Go 要点**：结构体未赋值的 `string` 字段零值为 `""`，不会报错但也不会自动填充——需要显式赋值。

## 六、配置层改动

### 6.1 `OssConfig` 新增字段

```go
type OssConfig struct {
    AccessKey  string `mapstructure:"access_key"`
    SecretKey  string `mapstructure:"secret_key"`
    Bucket     string `mapstructure:"bucket"`
    Region     string `mapstructure:"region"`
    Endpoint   string `mapstructure:"endpoint"`   // ← 新增：MinIO/S3 地址
    UseSSL     bool   `mapstructure:"use_ssl"`     // ← 新增：是否启用 TLS
    SignExpire int    `mapstructure:"sign_expire"`
}
```

### 6.2 `validateOssRequired()` — 启动前强校验

OSS 为强依赖，`Conf.Validate()` 调用 `validateOssRequired()`，任一必填项为空则启动失败：

```
oss.endpoint → oss.access_key → oss.secret_key → oss.bucket → oss.region
```

**设计意图**：fail fast，避免运行到上传接口才发现配置缺失。类比前端：`.env` 校验在应用启动时而非请求时。

## 七、Docker Compose 扩展 MinIO

### 7.1 新增服务

```yaml
minio:
  image: minio/minio:latest
  command: ["server", "/data", "--console-address", ":9001"]
  ports:
    - "9000:9000"   # S3 API
    - "9001:9001"   # Web 控制台
```

### 7.2 端口与用途

| 端口 | 用途 | 类比 |

| `9000` | S3 兼容 API（后端 SDK 连接用） | 类似 Redis 的 `6379` |

## 十、关键技术点总结

| # | 技术点 | 说明 |
|---|--------|------|
| 1 | **`minio.New(endpoint, &Options{})`** | S3 兼容 SDK，endpoint 为 `host:port`（无 scheme） |
| 2 | **`BucketExists` + `MakeBucket`** | 启动时确保 Bucket 存在，幂等操作 |
| 3 | **`PutObject(ctx, bucket, key, reader, size, opts)`** | `reader` 为 `io.Reader`，`file` 实现了此接口 |
| 4 | **`defer file.Close()`** | 确保文件句柄释放，Go 的 `try...finally` 等价物 |
| 5 | **`10 << 20`** | 位运算表达 10MB，Go 常见写法 |
| 6 | **`fmt.Errorf("...: %w", err)`** | `%w` 包装原始 error，调用方可 `errors.Is/As` 解包 |
| 7 | **返回 error vs `os.Exit`** | 底层返回错误，顶层决定退出——Go 错误处理惯例 |
| 8 | **`strings.TrimSpace` 校验** | 配置值可能带空格/换行，需 trim 后再判空 |
| 9 | **struct 零值** | `string` 零值为 `""`，`bool` 为 `false`——未赋值不报错但可能静默出错 |
| 10 | **`map[string]bool` 做白名单** | 比 slice 遍历 O(n) 更高效，查找 O(1) |

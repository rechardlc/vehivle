## 一、做了什么

### 1.1 新增迁移 `000005_create_param_templates_tables`

两张表：

```sql
param_templates       -- 参数模板（一级分类一对一绑定）
param_template_items  -- 参数模板项（模板一对多）
```

要点：
- `category_id` 外键 `REFERENCES categories(id) ON DELETE RESTRICT`——删分类前须先删/解绑模板
- `UNIQUE INDEX uq_param_templates_category_id`——保证一个一级分类**最多绑定一个模板**
- `param_template_items.field_type` 用 `CHECK` 约束 `text / number / single_select`
- `(template_id, field_key)` 联合唯一——同模板内字段键不能重复
- 排序索引 `(template_id, sort_order DESC, updated_at DESC)` 与项目通用排序规则一致

### 1.2 领域建模

**`model.ParamTemplate`**：
- `ID *string`（`omitempty`）——创建时 nil，由 DB `gen_random_uuid()` 回填
- `Items []ParamTemplateItem`——GORM `foreignKey:TemplateID; constraint:OnDelete:CASCADE`，handler 传入、repo 关联写入
- `Status enum.ParamTemplateStatus`——复用分类的启用/禁用 0/1 模式

**`model.ParamBody`**（更新 DTO）：
- 可选指针字段 `*string` / `*enum.ParamTemplateStatus`，与分类 PATCH 模式一致

**`model.ParamTemplateItem`**：
- `Required` / `Display`：`*int8`，`binding:"required,oneof=0 1"`——勿用 `bool`，与 DB SMALLINT 0/1 对齐
- `FieldType`：`binding:"required,oneof=text number single_select"`——API 层校验与 DB CHECK 一致

### 1.3 枚举

新增 `enum.ParamTemplateStatus`（`int8`，0/1），实现 `driver.Valuer` + `sql.Scanner`——与 `CategoryStatus` 同套路。

### 1.4 仓储（Repository）

`ParamTemplateRepo` 方法：

| 方法 | 说明 |
|------|------|
| `Create` | 直接 `db.Create(p)`，GORM 自动级联创建 Items（因 foreignKey 声明） |
| `Update` | **事务**内分三步：① `Omit("Items").Updates(p)` 更新模板本体；② 遍历 Items 逐条 `Save()`（有 ID 更新，无 ID 插入）；③ `DELETE ... WHERE id NOT IN (新 Items IDs)` 清理删除的项 |
| `Delete` | 物理删除模板（Items 由 `ON DELETE CASCADE` 级联） |
| `Detail` | 只查 `param_templates`，不加载 Items |
| `AllDetail` | `Preload("Items")` 联查模板 + 全部参数项 |
| `Count` | 全表计数（预留列表分页用） |

**核心学习点——Update 事务的三步模式**：

```go
r.db.WithContext(ctx).Transaction(func(tx *gorm.DB) error {
    // ① 更新主表（排除关联）
    tx.Model(p).Omit("Items").Updates(p)
    // ② 子表追加/更新（Save = upsert by PK）
    for i := range p.Items {
        p.Items[i].TemplateID = *p.ID
        tx.Save(&p.Items[i])
    }
    // ③ 删除前端已移除的子项
    tx.Where("template_id=? AND id NOT IN ?", *p.ID, keepIDs).Delete(&ParamTemplateItem{})
    return nil
})
```

这是 **"先更新、再同步子集、最后清理"** 的典型模式，处理一对多关联的 CRUD 非常通用。

前端类比：React 的 `useFieldArray`（react-hook-form）里增删改数组项后提交，后端需要同步持久化 + 清理已删项。

### 1.5 服务层

`ParamTemplateService` 薄包装：
- `Create`：直透 repo
- `Update`：先查 `Detail` 确认存在，再调 `repo.Update`
- `Detail` / `AllDetail`：直透 repo

### 1.6 Handler

`ParamTemplates` handler——四个端点：

## 二、GORM 一对多关联要点

### 2.1 Preload vs Join

| 方式 | SQL | 适用场景 |
|------|-----|---------|
| `Preload("Items")` | 主查询 + 子查询（两条 SQL） | 一对多，每条主记录可能有多条子记录 |
| `Joins("Profile")` | LEFT JOIN（一条 SQL） | 一对一关联 |

### 2.2 foreignKey 声明方向

```go
Items []ParamTemplateItem `gorm:"foreignKey:TemplateID"`
//                                          ↑ 子表的字段名
```

GORM 用**子表字段名**做 foreignKey（不是列名 `template_id`）。`constraint:OnDelete:CASCADE` 让 GORM `AutoMigrate` 时生成级联删除（但本项目用迁移管理 DDL，级联约束由 SQL 控制）。

### 2.3 Save 的 upsert 行为

```go
tx.Save(&item)
```

- **有 PK 值**：`UPDATE ... WHERE id = ?`
- **无 PK 值**：`INSERT`，回填 ID 到 struct

前端类比：Prisma 的 `upsert()` / Sequelize 的 `Model.upsert()`。

## 三、泛型函数入门

```go
func RequiredField[T ~string](field T) error {
    if field == "" {
        return fmt.Errorf("字段不能为空")
    }
    return nil
}
```

- `[T ~string]`：类型约束，`T` 的底层类型必须是 `string`（接受 `string` 本身或基于 string 的自定义类型）
- `~` 符号：Go 1.18+ 泛型语法，表示底层类型（underlying type）匹配
- 场景：handler 校验路径参数 `c.Param("id")`

前端类比：TypeScript 的 `function required<T extends string>(v: T)`。

## 四、这次学到的关键点

### 4.1 一对多 CRUD 的事务三步

更新主表 + 同步子表 + 清理已删子项——此模式在详情编辑场景（如车型参数、订单明细）中高频复用。

### 4.2 新模块接入组合根只需 3 行

`buildHandlers()` 加 repo → service → handler 三行，`Handlers` 结构体加一个字段，路由注册用 `r.handlers.Xxx.Method`——零耦合扩展。

### 4.3 `*int8` 而非 `bool`

数据库用 `SMALLINT 0/1`，Go 侧用 `*int8` + `binding:"oneof=0 1"`，避免 `bool` 的 JSON 零值 `false` 被 GORM 的 `Updates` 跳过。

### 4.4 Preload 字段名 ≠ 列名

`Preload("Items")` 里的 `"Items"` 是 Go struct 的字段名，不是表名也不是列名。

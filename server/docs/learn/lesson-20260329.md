## 今日目标

- 理解 **struct tag** 是「给某个库看的字符串」，**不是** Go 语法单独解释一层含义。
- 分清 **`json` / `form` / `gorm`** 各自被谁读取、用在哪一层协议。
- 明确 **`ShouldBindQuery`** 与 **`ShouldBindJSON`** 分别绑定 **URL query** 与 **JSON body**，不可混用场景。
- 区分 **GET 的 query 串** 与 **前端 `FormData`（多为 POST multipart）**。
- 理解 **Gin 中间件** 与 **`gin.HandlerFunc`**、**`router.Use`** 与 **路由级多 handler** 的选用。

## 一、struct tag 是什么

- 字段后反引号内如 `` `json:"parentId" form:"page" gorm:"column:created_at"` ``：**Go 编译器只当普通字符串**；是否生效取决于是否调用了会解析该 tag 名的库。
- **不同 tag 名 = 不同「读者」**：互不替代，可并列写在同一字段上。

---

## 二、常见 tag 与读者

| Tag | 典型读者 | 数据从哪来 |
|-----|----------|------------|
| **`json:"..."`** | `encoding/json`、Gin `ShouldBindJSON` | 请求/响应 **body** 里的 JSON |
| **`form:"..."`** | Gin `ShouldBindQuery`、`ShouldBind`（表单） | **URL query** `?key=value` 或 **x-www-form-urlencoded** / **multipart** body |
| **`gorm:"..."`** | GORM | **数据库** 表/列映射、忽略列等 |
| **`uri:"..."`** | Gin `ShouldBindUri` | **路径参数** `/users/:id` |
| **`binding:"..."`** | Gin validator | 与上述绑定常一起用 |

**记忆**：JSON 走 `json`；**列表筛选在 URL 上**走 `form` + `ShouldBindQuery`；**落库**走 `gorm`。

---

## 三、`ShouldBindQuery` 与 `form` 的关系

- **`ctx.ShouldBindQuery(&raw)`** 会把当前请求的 **query string** 填进结构体。
- 匹配依据是字段上的 **`form:"..."`** 标签名（与 `?page=1` 中的 `page` 对应）。
- **与 `json` 无关**：query 不是 JSON，不会用 `json` tag 去解析 URL。

本仓库示例：`handler/categories.go` 中 **`categoryListQueryParams`** 使用 `form:"keyword"`、`form:"page"` 等，供 **GET `/admin/categories?...`** 使用。

---

## 四、`ShouldBindJSON` 什么时候用

- 绑定 **HTTP body** 中的 **JSON**（通常 **POST/PATCH/PUT** + `Content-Type: application/json`）。
- **不能**用来替代「GET + 仅 query 参数」的列表接口：若客户端只发 `?page=1` 而无 body，`ShouldBindJSON` **读不到 query**，结构体多为零值。
- 若业务坚持「用 JSON 表达复杂查询条件」，常见做法是 **单独设计 POST 搜索接口**，body 为 JSON，再用 `ShouldBindJSON`——属于 **API 设计选择**，不是把现有 GET 列表直接改成 BindJSON。

---

## 五、GET 与「form」、前端 `FormData`

- **GET** 通常 **无 body**；筛选条件在 **URL query**（`key=value&...` 的编码风格与 HTML 表单字段类似，但数据在 **URL** 上）。
- **Gin 的 `form` tag + `ShouldBindQuery`**：对应的是 **query 参数名**，不是「浏览器 POST 整表提交」的同一件事。
- **`new FormData()`** 多用于 **POST**，常配合 **`multipart/form-data`**（尤其含文件）；**列表 GET** 常规做法是 **`params` / `URLSearchParams`** 拼 query，**不是**把 `FormData` 塞给 GET body（行为与缓存/代理不一致，也不常见）。

**结论**：GET 列表的「像表单」指的是 **query 键值对**；**`FormData` 是另一套（多为 POST body）**，与 `ShouldBindQuery` 不是同一用法。

---

## 六、与 `gorm` / `json` 并存

同一字段可同时需要 JSON 与 DB 映射，例如：

```go
CreatedAt time.Time `json:"createdAt" gorm:"column:created_at"`
```

含义：**JSON** 里叫 `createdAt`；**表**里列名 `created_at`；二者独立。

## 七、Gin 中间件（`gin.HandlerFunc`）

### 7.1 类型是什么

- **`gin.HandlerFunc`** 即 **`func(*gin.Context)`**。
- **业务 handler**（如 `categoriesHandler.List`）与 **中间件** **类型相同**；是否叫「中间件」主要看 **怎么挂到链上**、是否调用 **`c.Next()`** 放行后续逻辑。

### 7.2 `Next` 与 `Abort`

- **`c.Next()`**：继续执行**后面的**中间件与最终 handler。
- **`c.Abort()`**：**不再**往后执行（常用于已写错误响应、校验失败）。

### 7.3 两种挂法：`Use` vs 单路由多 handler

| 方式 | 典型写法 | 作用范围 |
|------|----------|----------|
| **组 / 引擎级** | `router.Use(mw)` 或 `group.Use(mw)` | 该组（或引擎）上**之后注册**的路由**都会**经过 `mw`（日志、鉴权、CORS 等）。 |
| **单路由** | `group.GET("/path", mw, handler)` | **仅这一条**路由先执行 `mw`，再执行 `handler`（例如只对列表 GET 做 query 键白名单）。 |

- **`GET` / `POST` 等**可接收**多个** `HandlerFunc`，按顺序执行；中间件内需 **`c.Next()`** 才会进入下一环。

### 7.4 需要「配置」时：工厂函数返回 `HandlerFunc`

- 若中间件依赖**参数**（如允许的 query 键列表），常写成 **外层函数返回内层 `func(c *gin.Context)`**。
- 本仓库示例：**`middleware.ValidateParams(allowFields []string) gin.HandlerFunc`** —— 外层把 `allowFields` 建成 `map`，内层遍历 `c.Request.URL.Query()` 的 key，不在白名单则 **`FailParam` + `c.Abort()`**，否则 **`c.Next()`**。

### 7.5 路由中的实际用法

- 分类列表：**`categories.GET("", middleware.ValidateParams(categoryList), categoriesHandler.List)`**（见 `internal/transport/http/router/router.go`）。
- 只对 **GET 列表** 校验 query 键，**避免** `POST`/`PUT`/`DELETE` 被同一套「仅列表需要」的规则误伤。

---

## 复盘

- **标签混乱** 往往来自把「一个字段」想成只有一种写法；实际是 **query / JSON / DB** 三条通道，各用各的 tag。
- **列表 GET** → **`ShouldBindQuery` + `form`**；**写 body** → **`ShouldBindJSON` + `json`**；**落库** → **`gorm`**。
- **中间件**：类型都是 **`HandlerFunc`**；**批量**用 **`Use`**，**单条路由**用 **`GET(..., mw, h)`**；放行用 **`Next`**，失败用 **`Abort`**。

## 一、这次为什么要重构 `system_settings`

旧表是 **key-value + JSONB** 思路：一行一个 `setting_key`，值放进 `setting_value`。它的优点是“灵活”，但对当前 V1 的**固定全局配置**并不友好：

- **必填约束难做**：例如 `company_name`、`default_price_mode` 这类字段，key-value 很难直接做 `NOT NULL` + 枚举检查。
- **读取心智复杂**：前端/后端都要先“组装对象”，不像单行配置表那样天然就是一个 settings 对象。
- **校验分散**：字段规则更容易散落在业务代码里，而不是直接落在表结构里。

这次改成**单行表**后，数据库本身就能表达“系统设置只有一份配置对象”的语义。

类比前端：

- **旧设计**像把页面表单拆成 `Map<string, unknown>` 存 localStorage，再在运行时拼回对象。
- **新设计**像直接维护一个 `const settings: SystemSettings`，类型和字段约束都更直观。

## 二、迁移 `000004_alter_system_settings`

- 文件：`migrations/000004_alter_system_settings.up.sql` / `.down.sql`
- 做法：直接 `DROP TABLE IF EXISTS system_settings` 后重建为**单行表**
- 关键约束：
  - `id INTEGER PRIMARY KEY DEFAULT 1 CHECK (id = 1)`：保证全表最多一行
  - `company_name TEXT NOT NULL`：公司名必填
  - `default_price_mode TEXT NOT NULL CHECK (...)`：只允许 `show_price` / `phone_inquiry`
  - 客服电话、客服微信、免责声明、默认分享标题/图允许 `NULL`

这一步的核心不是“字段变多了”，而是**表语义变了**：从“动态配置字典”变成“固定结构的配置对象”。

## 三、后端链路怎么接通

### 3.1 Domain：`model.SystemSetting`

- 新增 `internal/domain/model/system.go`
- 用一个 struct 映射整张 `system_settings` 表
- 必填字段用值类型：`CompanyName string`、`DefaultPriceMode string`
- 可选字段用指针：`*string`

前端类比：

- **TypeScript 可选字段**：`customerServicePhone?: string`
- **Go 可空字段**：`CustomerServicePhone *string`

区别在于：

- TS 的 `undefined` 只是运行时值之一
- Go 里用 `*string == nil` 明确表示“数据库可为 NULL”

### 3.2 Repository：把“唯一一行配置”封装起来

- `Detail(ctx)`：查详情
- `Create(ctx, m)`：首次创建时强制 `m.ID = 1`
- `Update(ctx, updates)`：固定 `WHERE id = 1`
- `Exists(ctx)`：判断是否已创建

这层的重点是把“只有一行”藏在仓储内部，Handler/Service 不必关心主键怎么传。

前端类比：

- 像把 `localStorage.getItem("system-settings")` / `setItem(...)` 封装进一个 `settingsStore`
- 业务层只知道“读设置”和“改设置”，不用每次都自己拼 key

### 3.3 Service：把“创建一次、之后只能更新”的规则收口

- `Create()`：如果已存在，报错“系统配置已存在，请使用更新接口”
- `Update()`：如果不存在，报错“系统配置不存在，请先创建”

这里是典型 Go 分层思路：

- Repository 负责“怎么查 / 怎么写”
- Service 负责“什么时候允许查 / 写”

前端类比：

- Repository 更像 `api.ts`
- Service 更像你在 React Query / Zustand 外再包一层业务 hook，专门处理流程规则

### 3.4 Handler：把 HTTP 契约变成前端可直接消费的 JSON

新增 `/api/v1/admin/system-settings`：

- `GET /admin/system-settings`：详情；如果还没创建，返回 `data: null`
- `POST /admin/system-settings`：首次创建
- `PUT /admin/system-settings`：更新已有配置

额外做了两件很实用的事：

1. **请求体用 camelCase**
   例如 `defaultPriceMode`、`customerServicePhone`，方便管理端直连
2. **响应体补 `defaultShareImageUrl`**
   数据库存的是 `storage_key`，返回前由 OSS 客户端拼成可预览 URL，前端能直接回显图片

## 四、管理端为什么要做价格模式映射

数据库/后端枚举：

- `show_price`
- `phone_inquiry`

管理端 UI 枚举：

- `msrp`
- `negotiable`

因此 `admin/src/api/settings.ts` 加了双向映射：

- **后端 → 前端**：`show_price` → `msrp`
- **前端 → 后端**：`negotiable` → `phone_inquiry`

这一步非常像前端常见的 **DTO adapter**：

- 后端返回的是“接口契约语言”
- 页面表单使用的是“组件/UI 语言”

如果不加这一层，UI 文案、Select 值和数据库枚举会缠在一起，后续很容易改坏。

## 五、这次学到的几个关键点

### 5.1 单行表不是偷懒，而是更贴近业务事实

`system_settings` 本质上不是列表，也不是多租户配置集合，而是**整套系统只有一份的全局配置对象**。用单行表反而最简单、最难写错。

### 5.2 `NULL` 和空字符串不是一回事

表结构注释里写的是：

- 客服电话未配置则前端隐藏
- 客服微信未配置则前端隐藏

当前实现里：

- Create 可传 `nil`
- Update 若传空字符串，会写入 `""`，不会自动转回 `NULL`

这意味着“没填”和“清空成空串”在数据库里目前不是完全同一语义。后续如果小程序/公开接口严格依赖 `NULL` 判断隐藏，需要再统一口径。

### 5.3 `storage_key` 和 `public URL` 要分开

- 表里存 `default_share_image`：本质是对象键
- 接口返回 `defaultShareImageUrl`：给前端预览

前端类比：

- 数据层存资源 `id`
- 展示层再派生出 `previewUrl`

这样换域名、换 CDN、换桶策略时，不用回写业务表。

## 六、当前范围边界与仍未完成的部分

按 V1 约束，系统设置属于**后台必做**，主要承载：

- 公司名称
- 客服电话 / 微信
- 默认价格模式
- 默认免责声明
- 默认分享标题 / 图

但当前代码状态仍有两个边界要记住：

1. **目前只完成后台管理接口与页面**
   公开端/小程序侧读取系统设置的聚合接口还没补出来
2. **价格模式只是全局默认值**
   车型维度仍可能覆盖；若全局显示价格，但某车型没填 `msrp_price`，V1 规则仍应回落为“电话询价”

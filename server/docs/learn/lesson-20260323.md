## 一、问题现象

前端 `axios` 请求 `GET /api/v1/admin/vehicles`，Network 面板出现两条请求：

1. `GET /api/v1/admin/vehicles` → **301 Moved Permanently**
2. `GET /api/v1/admin/vehicles/` → 200 OK（浏览器自动跟随重定向）

页面功能正常但多了一次无意义的网络往返，且 301 在 POST/PUT 场景下会导致请求方法被改变（POST → GET），属于潜在的生产级 bug。

## 二、排障过程与知识点

### 2.1 第一层：Gin 的 `RedirectTrailingSlash` 机制

Gin 引擎默认 `RedirectTrailingSlash = true`。当请求路径与已注册路由仅差一个尾部 `/` 时，Gin 自动返回 301 重定向。

| 设置 | 行为 |
|------|------|
| `RedirectTrailingSlash = true`（默认） | `/vehicles` → 301 → `/vehicles/` |
| `RedirectTrailingSlash = false` | 不重定向，严格匹配 |

**为什么默认开启？** 源自传统 Web 服务器（Apache/Nginx）时代的约定：`/path` 表示文件，`/path/` 表示目录。Gin 沿袭此传统做"容错"。

**企业 API 项目应关闭的三个理由：**

1. **多一次网络往返**：前后端分离项目中 API 应直接返回 JSON，不应重定向。
2. **POST/PUT 的 301 陷阱**：HTTP 规范允许浏览器将 301 重定向后的 POST 改为 GET，请求体丢失。
3. **接口契约模糊化**：自动纠错会掩盖前端路径拼接错误，增加排障成本。

**修复**（`bootstrap.go`）：

```go
r := gin.New()
r.RedirectTrailingSlash = false
```

### 2.2 第二层：修改后仍然 301——air 编译失败的静默降级

设置 `RedirectTrailingSlash = false` 后刷新页面，**301 依然存在**。

终端中 `air` 的输出看起来正常（`running...` → `bootstrap ready` → `Listening on :9999`），但仔细翻看历史输出发现：

```
internal\bootstrap\bootstrap.go has changed
building...
Process Exit with Code: 1
# vehivle/internal/domain/rule
internal\domain\rule\vehicle_publish.go:21:21: invalid operation: v.CategoryID == "" (mismatched types *string and untyped string)
failed to build, error: exit status 1
running...
time=2026-03-23T08:57:28.139Z level=INFO msg="bootstrap run" ...
```

**关键发现**：`air` 编译失败后**不会停止服务**，而是继续运行上一次编译成功的旧二进制。紧接着旧二进制启动时输出大量路由注册日志，把编译错误刷出屏幕可视区域。

**air 编译失败的行为链：**

```
1. 检测到 .go 文件变动
2. building...                    ← 尝试编译
3. 编译错误（2-3 行）              ← 真正的报错，很快被后续输出刷走
4. failed to build, error: ...    ← 一行提示
5. running...                     ← 启动旧二进制（看起来像正常启动）
6. bootstrap ready                ← 路由注册 + 启动日志（20+ 行）
7. Listening on :9999             ← 一切看起来正常
```

**对比前端**：Vite/Webpack 编译失败时会在浏览器弹出全屏红色错误遮罩，不可能忽略。`air` 没有这种机制。

**排障建议**：改了 Go 代码后行为没变化时，**第一步手动验证编译**：

```bash
go build ./cmd/api
```

没输出 = 编译通过；有报错 = `air` 跑的是旧代码。

### 2.3 第三层：编译错误的根因——`*string` 指针类型不匹配

`Vehicle.CategoryID` 之前从 `string` 改为 `*string`（对齐数据库 `UUID NULL`），但两处代码未同步更新：

| 文件 | 旧代码（编译失败） | 修复后 |
|------|-------------------|--------|
| `domain/rule/vehicle_publish.go` | `v.CategoryID == ""` | `v.CategoryID == nil \|\| *v.CategoryID == ""` |
| `handler/vehicles.go` | `CategoryID: c.PostForm("category_id")` | `CategoryID: strPtr(c.PostForm("category_id"))` |

**`strPtr` 辅助函数**（`handler/vehicles.go`）：

```go
func strPtr(s string) *string {
    if s == "" {
        return nil
    }
    return &s
}
```

空串返回 `nil`，对齐数据库 `NULL` 语义。

### 2.4 Gin radix tree 与 Group 的尾斜杠行为

旧二进制中路由注册调试输出：

```
GET    /api/v1/admin/vehicles/   → 带尾斜杠（有子路由 :vehicle_id 等）
GET    /api/v1/admin/categories  → 无尾斜杠（无子路由）
GET    /api/v1/public/vehicles   → 无尾斜杠（无子路由）
```

**规律**：当 `Group("/vehicles")` + `GET("")` 配合子路由（如 `PUT /:vehicle_id`、`POST /batch-status` 等）时，Gin 的 radix tree 会将 `/vehicles` 视为中间节点，将 `GET ""` 的处理器挂到 `/vehicles/`（带尾斜杠）上。

新二进制（编译修复后）当前 `router.go` 仅有 `GET ""`、`POST ""`、`PUT /:vehicle_id`、`DELETE /:vehicle_id`（无 `batch-status` 等），路由注册为：

```
GET    /api/v1/admin/vehicles    → 无尾斜杠 ✅
```

## 四、经验总结

### 4.1 排障思维模型

```
表面现象（301 重定向）
    ↓ 为什么？
Gin RedirectTrailingSlash 默认开启
    ↓ 改了为什么没生效？
air 编译失败，跑的是旧二进制
    ↓ 为什么编译失败？
CategoryID 从 string 改为 *string，两处代码未同步
    ↓ 修复后
编译通过 → air 重启新二进制 → 301 消失
```

**核心教训**：当修改"没有生效"时，先验证修改是否已经编译进运行的二进制，而不是继续在代码层面找原因。

### 4.2 Go 指针类型修改的连锁影响

字段类型从 `T` 改为 `*T` 时，所有使用该字段的地方都需要同步更新：

| 操作 | `string` 写法 | `*string` 写法 |
|------|--------------|----------------|
| 判空 | `s == ""` | `s == nil \|\| *s == ""` |
| 赋值 | `Field: value` | `Field: &value` 或 `Field: strPtr(value)` |
| 读取 | `s` | `*s`（需先判 nil） |
| JSON 序列化 | `"field": ""` | `"field": null`（nil 时） |

**建议**：改 model 字段类型后，用 `go build ./cmd/api` 一次性暴露所有编译错误，逐一修复。

### 4.3 air 热重载使用备忘

| 场景 | air 行为 | 开发者应做 |
|------|---------|-----------|
| 编译成功 | 杀旧进程 → 启新二进制 | 正常使用 |
| 编译失败 | 打印错误 → `running...` → 启旧二进制 | 搜索 `failed to build` 或手动 `go build` 验证 |
| 新增依赖 | 可能需要先 `go mod tidy` | 先 tidy 再让 air 重编译 |
| 配置文件变动 | air 仅监听 `.go`/`.html` 等 | 手动重启 air |

## 五、各框架尾斜杠对比（扩展认知）

| 框架 | 默认行为 | 适用场景 |
|------|---------|---------|
| **Gin** | `RedirectTrailingSlash = true` | 通用 Web（兼顾页面 + API） |
| **Echo** | 无自动重定向 | API 优先 |
| **Fiber** | `StrictRouting = false`（类似 Gin） | 通用 Web |
| **Express (Node)** | 严格匹配，无自动重定向 | API 优先 |
| **Spring Boot** | `useTrailingSlashMatch = false`（5.3+） | 已废弃自动匹配 |

趋势：**越现代、越偏 API 的框架，越倾向于严格匹配、不做自动重定向。**

## 复盘

- **排障路径**：表面问题（301）→ 配置修复（`RedirectTrailingSlash`）→ 发现修改未生效 → 定位编译失败 → 修复类型错误 → 问题解决。核心教训是**先验证编译，再查逻辑**。
- **air 的坑**：编译失败后静默降级到旧二进制，且大量启动日志刷走错误信息。养成 `go build ./cmd/api` 手动验证的习惯。
- **指针类型连锁**：`string → *string` 的 model 变更需全局排查赋值、比较、解引用三类用法。
- **Gin 路由内部机制**：`Group` + 子路由会影响 radix tree 节点结构，可能导致尾斜杠偏移。
- **明日第一件事**：继续推进车型 CRUD（`Create` handler 已接通 Service；`Update`/`Delete` handler 仍为占位）。

## 六、API JSON 命名统一（camelCase）+ 分类模块 CRUD

### 6.1 前后端 JSON 字段命名统一

**问题**：后端 `model.Category` 与 `model.Vehicle` 的 `json` tag 使用 `snake_case`（如 `parent_id`、`sort_order`），而 admin 前端使用 `camelCase`（如 `parentId`、`sortOrder`）。前端需要一整套映射层（`CategoryApiRow`/`VehicleApiRow` + `mapRowToListItem`）做转换，两边重复维护、联调成本高。

**决策**：全站 API JSON 统一 **camelCase**，与 TypeScript 命名习惯一致。

**涉及文件与改动**：

| 文件 | 改动 |
|------|------|
| `model/category.go` | `json:"parent_id"` → `json:"parentId"`，所有字段统一 camelCase |
| `model/vehicle.go` | `json:"category_id"` → `json:"categoryId"`，所有字段统一 camelCase |
| `pkg/response/response.go` | `json:"request_id"` → `json:"requestId"` |
| `handler/vehicles.go` | `PostForm` 改为 `ShouldBindJSON` + camelCase 请求体结构 |
| `handler/user.go` | `gin.H` 中的 `user_id`/`created_at` 等改为 `userId`/`createdAt` |
| `router/router.go` | health 响应 `request_id` → `requestId` |
| `admin/src/types.ts` | `ApiResponse.request_id` → `requestId` |
| `admin/src/api/categories.ts` | 删除 `CategoryApiRow` 与 `mapCategoryRows` 映射层，直接用 `Category` 类型 |
| `admin/src/api/vehicles.ts` | 删除 `VehicleApiRow` 与 `mapRowToListItem` 映射层，`create` 从 FormData 改为 JSON |

**核心知识点**：

- Go struct 的 `json` tag 控制 `encoding/json` 序列化的 key 名，与 GORM 列映射互不干扰（GORM 默认按 Go 字段名蛇形转换映射列名，如 `ParentID` → `parent_id`）。
- 统一后前端**不再需要任何字段名映射层**，接口响应可直接匹配 TS 类型。

### 6.2 分类状态改为 0/1

**问题**：`categories.status` 最初设计为 `TEXT('enabled'/'disabled')`，但企业开发中数据库列更常用 `SMALLINT(0/1)` — 省空间、索引友好、BI 友好。

**改动链**：

| 层 | 变更 |
|----|------|
| 迁移 `000002` | `status TEXT` → `SMALLINT NOT NULL DEFAULT 1 CHECK (status IN (0, 1))` |
| `enum.CategoryStatus` | `type CategoryStatus string` → `type CategoryStatus int8`（0=禁用, 1=启用） |
| handler | `ParseCategoryStatus` 支持 `"0"`/`"1"` 解析 |
| admin | `CategoryStatus = "enabled" \| "disabled"` → `CategoryStatus = 0 \| 1` |
| DB ↔ Go | 实现 `Value()`/`Scan()` 接口做 `int8 ↔ int64` 读写 |
| JSON | 不再自定义 `MarshalJSON`/`UnmarshalJSON`，直接序列化为数字 0/1 |

**经验**：一个枚举的存储格式变更会穿透 **迁移 → domain enum → handler → admin types → 页面渲染** 全链路。

### 6.3 Go struct embedding（匿名嵌入）消除 DTO 与 Entity 的重复

**问题**：`CategoryCreateInput`（创建请求体）与 `Category`（领域实体）有 5 个完全相同的字段和 json tag。

**解法**：Go 的匿名嵌入（类似 TS 的 `interface extends`）。

```go
type CategoryCreateInput struct {
    Name      string              `json:"name"`
    ParentID  *string             `json:"parentId"`
    Level     int                 `json:"level"`
    Status    enum.CategoryStatus `json:"status"`
    SortOrder int                 `json:"sortOrder"`
}

type Category struct {
    ID string `json:"id"`
    CategoryCreateInput             // 匿名嵌入，字段提升到外层
    IconMediaID *string             `json:"iconMediaId,omitempty"`
    CreatedAt   time.Time           `json:"createdAt"`
    UpdatedAt   time.Time           `json:"updatedAt"`
    ParentName  string              `json:"parentName,omitempty" gorm:"-"`
}
```

| 特性 | 行为 |
|------|------|
| **JSON** | `encoding/json` 把嵌入字段提升到外层，输出 `{ id, name, parentId, ... }` 扁平 JSON |
| **GORM** | 嵌入字段按提升后的字段名映射到表列（与之前等价） |
| **代码访问** | 仍写 `cat.Name`、`cat.ParentID`，无额外前缀 |
| **构造** | `Category{ ID: ..., CategoryCreateInput: body }` 一行完成 |

**适用边界**：

- ✅ 入参与实体字段**高度一致**，只差几个只读字段。
- ⚠️ 当 Create/Update/Patch 参数差异大，或需要多版本 API 时，分开写更清晰。
- ⚠️ 大团队中需要所有人理解 Go 的 promotion 规则，否则可能困惑。

### 6.4 分类列表的 parentName 填充

后端 `Category` 新增虚拟字段 `ParentName`（`gorm:"-"` 不映射表列），在 handler 的 `List` 方法中根据 `id → name` 映射填充。admin 直接使用 `parentName` 渲染「父级分类」列，不再需要前端推导。

## 七、今日经验总结

| # | 经验 |
|---|------|
| 1 | **前后端 JSON 约定**：企业项目应在契约里固定一种命名风格（camelCase 或 snake_case），全站统一。Go 的 `json` tag 与 GORM 列名独立，改 tag 不影响 DB。 |
| 2 | **状态枚举设计**：库里用 `SMALLINT(0/1)`、API 也用数字是最简洁的全链路方案；若需要 API 可读性，可在 Go 层用自定义 `MarshalJSON`/`UnmarshalJSON` 做数字↔字符串映射。 |
| 3 | **已执行的迁移不要改**——用新版本 `ALTER`；若纯本地开发，可以改后删库重建。 |
| 4 | **Go struct embedding**：消除 DTO 与 Entity 重复的惯用手法，但要团队一致认可；大项目中更常见显式分离。 |
| 5 | **handler 的请求绑定**：`ShouldBindJSON` 绑定到专用 Input 类型，而非完整 Entity，防止客户端伪造 `id`/时间戳等只读字段。 |
| 6 | **`gorm:"-"`**：标记虚拟字段不写入数据库，适合列表展示时的派生数据（如 `parentName`）。 |

## 今日目标

- 理解 **部分更新（PATCH 语义）** 在 Go 里如何用类型表达「字段未出现」与「字段出现」。
- 区分 **`Name`（`*string` → `string`）** 与 **`ParentID`（`*string` → `*string`）** 合并时的赋值写法。
- 将创建与更新后的 **领域校验** 收敛到 **`validateResolvedCategory(model.CategoryCreateInput)`**，避免重复 `if`。

---

## 一、背景：为什么不能用「嵌入 `CategoryCreateInput`」做 Update 请求体？

`CategoryCreateBody` 创建场景使用 **`CategoryCreateInput`**（值类型字段）是合理的：每个字段都有明确值。

若 **`CategoryUpdateBody` 直接嵌入 `CategoryCreateInput`**，JSON **未出现**的键在反序列化后会变成 **Go 的零值**：

| 字段 | 未出现在 JSON 中的含义（期望） | 实际绑定结果（值类型） |
|------|-------------------------------|------------------------|
| `status` | 不改库里的状态 | `0`（与「停用」合法值冲突） |
| `level` | 不改层级 | `0`（与「合法 level 为 1/2」混淆） |
| `sortOrder` | 不改排序 | `0`（与「必须 > 0」校验混淆） |
| `name` | 不改名称 | `""`（曾用 `if body.Name != ""` 可凑合，但与显式空串无法区分） |

因此 **Update 专用** 的 `CategoryUpdateBody` 改为 **各字段均为可选指针**（`omitempty`），语义对齐前端 **`Partial<Category>`**：**仅非 `nil` 的指针字段表示本次提交**。

---

## 二、`CategoryUpdateBody` 结构（节选）

```go
// CategoryUpdateBody PATCH 部分更新：仅非 nil 指针字段表示本次提交（omit 的键不覆盖库内原值）。
type CategoryUpdateBody struct {
	Name      *string              `json:"name,omitempty"`
	ParentID  *string             `json:"parentId,omitempty"`
	Level     *int                `json:"level,omitempty"`
	Status    *enum.CategoryStatus `json:"status,omitempty"`
	SortOrder *int                `json:"sortOrder,omitempty"`
}
```

| JSON 形态 | `body.Status`（示例） | 含义 |
|-----------|------------------------|------|
| 无 `status` 键 | `nil` | 不修改 `existing.Status` |
| `"status": 0` | 非 `nil`，值为 `0` | 明确改为停用 |
| `"status": 1` | 非 `nil`，值为 `1` | 明确改为启用 |

若 `Status` 用值类型 `int8`：**未传**与**传 0** 在内存中无法区分。

## 三、Handler 合并逻辑：指针 → `existing`

合并后再调用 **`validateResolvedCategory(existing.CategoryCreateInput)`**，保证「创建」与「更新后的最终态」共用同一套规则。

**易错点（已踩坑）**：曾错误写成「仅当 `status` **不是** 0 且 **不是** 1 时才写入」——会导致合法值 **永远不写入**。正确做法是：**若 `body.Status != nil`，则 `existing.Status = *body.Status`**，再由 **`validateResolvedCategory`** 校验最终状态是否 ∈ {0,1}。

---

## 四、`Name` 与 `ParentID` 赋值为何写法不同？

领域模型里（`CategoryCreateInput`）：

| 字段 | 类型 | 含义 |
|------|------|------|
| `Name` | `string` | 必有名称（空串非法，由校验拦截） |
| `ParentID` | `*string` | 可选外键；`nil` 表示无父级 |

PATCH 体里二者都是 `*string`（可选「是否本次提交」），合并到 `existing` 时：

```go
if body.Name != nil {
	existing.Name = *body.Name   // *string → string，需解引用
}
if body.ParentID != nil {
	existing.ParentID = body.ParentID // *string → *string，拷贝指针
}
```

- **`Name`**：实体侧是 **`string`**，必须把指针解引用成字符串。
- **`ParentID`**：实体侧已是 **` *string`**，表示库里可空 UUID，**指针赋指针**即可，无需 `*body.ParentID`。

## 五、`validateResolvedCategory` 的职责边界

- **入参**：`model.CategoryCreateInput`（名称、排序、层级、状态、父级等 **已确定** 的一套值）。
- **调用方**：
  - **Create**：绑定 `CategoryCreateInput` 后直接校验。
  - **Update**：将 `body` 合并进 `existing` 后，校验 **`existing.CategoryCreateInput`**（嵌入字段提升后的访问方式不变）。

避免在 handler 里重复多段相同的 `if` + `FailBusiness`。

## 复盘

- **PATCH 在 Go 里的常用做法**：可选字段用 `*T`；数字/status 合法值含 `0` 时，指针几乎是必选项。
- **同一指针类型赋值**（`ParentID`）与 **指针到值**（`Name`）不要混用解引用。
- **合并后再校验**：避免只对「请求体」校验而忽略「库内旧值 + 部分字段」的最终一致性。

## 明日可延续

- 车型 `Update`/`Delete` 若也支持部分更新，可复用「指针 PATCH 体 + 合并 + 单点校验」模式。
- 若需区分 JSON **`null`** 与 **omit**（如对 `parentId`），需自定义 `UnmarshalJSON` 或引入中间 DTO，当前与多数项目一样接受「二者合并为一种语义」。

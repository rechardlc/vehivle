## 一、`migrations/` 与 GORM：两套机制

| 机制 | 作用 | 运行时是否被 GORM 读取 |
|------|------|------------------------|
| **`server/migrations/*.sql`** | 由 **golang-migrate**（`cmd/migrate`）执行，**创建/变更** PostgreSQL 中的表、索引、外键 | **否**。迁移文件不参与 ORM 解析 |
| **GORM** | 连接已有数据库，按 **struct + tag + `TableName()`** 生成 `SELECT`/`INSERT`/`UPDATE` | 只认 **Go 类型**，不认迁移目录 |

**结论**：删掉 `migrations/` 目录不会让 GORM「不认表」——表名仍由 struct 推断；但若 **新环境没有执行过迁移**，库里根本没有表，GORM 仍会按 struct 去访问 **不存在的表**，从而报错。**迁移是 schema 的单一事实来源（SSOT）之一；GORM 是访问层的映射规则。**

---

## 二、表名：`Category` 如何对应 `categories`

1. **默认行为（未实现 `TableName()` 时）**：GORM 使用 **NamingStrategy**，将结构体名 **英文复数化** 得到表名，例如 `Category` → **`categories`**（不是字符串包含关系，而是规则推断）。
2. **表名不规则时**（如 `categories_bak`、`tbl_category`）：默认推断会错，需 **显式指定**：
   - **`func (Category) TableName() string { return "categories" }`**（或指针接收者 `*Category`），与 `CREATE TABLE` 中的名字一致。
3. **GORM 约定**：方法名必须为 **`TableName`**，返回 **`string`**；值接收者与指针接收者均可被 GORM 识别。

本仓库在 `internal/domain/model/category.go` 中为 **`categories`** 显式实现 `TableName()`，与 `migrations/000002_add_categories_table.up.sql` 对齐，**避免仅依赖默认复数规则**。

## 三、列名：字段 → 列不是「猜」，是命名策略

- **默认**：结构体字段 **`CreatedAt`** → 列 **`created_at`**（蛇形小写），与 PostgreSQL 常见列名一致。
- **显式**：`gorm:"column:created_at"` 与默认结果相同时，行为不变，但 **与迁移 SQL 对照更清晰**，全局修改 `NamingStrategy` 时也更稳。
- **非列字段**：`gorm:"-"` 表示 **不参与持久化**（如列表展示的 **`ParentName`**），表中无此列。

**注意**：`json:"..."` 与 `gorm:"..."` 独立；改 JSON 名不改变数据库列映射。

## 四、struct 必须包含表的全部字段吗？

**不必须。**

- **少写字段**：查询时未映射的列不会进 struct；写入时未出现的字段通常不会出现在对应语句中（仍须注意 `Save` 等全量语义时的零值问题，必要时用 `Select`/`Omit`）。
- **多列、只读部分**：可以只定义业务需要的字段。
- **长期维护同一表的主路径 CRUD**：建议 **字段与迁移对齐**，或显式 `column` / `gorm:"-"`，减少漏读、误更新。

---

## 五、同一文件中的多个类型：`Category` + `CategoryCreateInput` + `CategoryUpdateBody` + `CategoryListQuery`

**合理**，尤其在中小规模项目中：

- 类型均围绕 **「分类」聚合**，内聚高、文件短，便于对照。
- **`Category` 匿名嵌入 `CategoryCreateInput`**：创建可写字段与实体共用一套定义，避免两处漂移（详见 [lesson-20260323](./lesson-20260323.md)、[lesson-20260325](./lesson-20260325.md)）。
- 若团队要求 **domain 不含 HTTP 契约**：可将带 `json` tag 的 DTO 挪到 `transport`；当前放在 `domain/model` 是常见折中。

**可选进阶**：复杂遗留库可拆 **PO（持久化对象）** 与 **领域实体**，在 Repository 内转换；非必须。

## 六、企业实践速查

| 场景 | 常见做法 |
|------|----------|
| 新表、命名规范 | 迁移列名 **snake_case**，Go 字段 **PascalCase**，依赖默认映射 + 迁移对齐 |
| 表名/列名怪异 | **`TableName()`** + **`gorm:"column:..."`** |
| 版本化 schema | **迁移工具** 管理 DDL；生产慎用 `AutoMigrate` 作为唯一来源 |
| 与前端 JSON | **camelCase** 在 `json` tag；数据库仍 **snake_case** |

---

## 复盘

- **迁移管「有没有这张表」**；**GORM 管「用哪张表、哪些列」**——二者通过 **人工约定** 对齐，不是自动读迁移文件。
- **`TableName()`** 是 GORM 识别的固定方法名，属于 **类型方法**，不是包级函数。
- **`Category` 单文件多类型** 在本项目阶段 **可接受**；按文件长度与团队分层规则再决定是否拆分。

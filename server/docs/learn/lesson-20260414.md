# 2026-04-14 全链路关系审计与企业级解法

> 本文以当前 `server` 真实代码为准，回答三个问题：后端链路是否完整、当前缺陷在哪里、企业项目通常如何治理这些链路问题。

---

## 一、范围边界复述

本次审计范围是 `server` Go/Gin 后端，不包含前端页面实现细节，也不直接改业务代码。

V1 后端目标是支撑两/三轮车电子展厅：

- 管理端：登录、车辆管理、分类管理、系统设置、媒体上传、参数模板维护。
- 公开端：小程序/访客读取已发布车辆与展厅展示数据。
- 基础设施：PostgreSQL、MinIO/S3、JWT Cookie、统一响应、请求日志、迁移脚本。

审计依据：

- 启动链路：`cmd/api/main.go -> configs.Load -> bootstrap.Run -> router.Register`。
- 请求链路：`middleware -> handler -> service -> repository -> DB/OSS -> response`。
- 数据链路：`migrations -> domain/model -> repository SQL/GORM -> handler response`。
- 学习文档：`server/docs/learn/README.md`、`progress.md`、`lesson-*.md`。

## 二、当前全链路总图

### 1. 启动链路

```text
cmd/api/main.go
  -> configs.Load()
     -> .env / .env.{env}
     -> configs/conf.{env}.yaml
     -> VEHIVLE_* 环境变量覆盖
     -> cfg.Validate()
  -> bootstrap.New(cfg)
  -> bootstrap.Run()
     -> gin.New()
     -> Gin 全局中间件
        -> gin.Logger()
        -> gin.Recovery()
        -> logger.RequestID()
        -> logger.AccessLog()
     -> postgres.Open()
     -> postgres.Ping()
     -> ossConnPool()
        -> MinIO client
        -> BucketExists / MakeBucket
        -> ObjectPublicURL 基址
        -> 可选 public read policy
     -> buildHandlers()
        -> repo
        -> service
        -> handler
     -> router.Register()
  -> r.Run(":port")
```

结论：启动链路结构是清晰的，已经具备企业项目常见的组合根模式。当前不足是没有优雅关闭、没有 readiness/liveness 分层、OSS 被设为强依赖后会导致媒体服务不可用时整个 API 无法启动。

### 2. HTTP 请求链路

```text
Client
  -> Gin Engine
  -> RequestID / AccessLog / Recovery
  -> Route Group
     -> public: no auth
     -> admin: JWTAuth
     -> route-level ValidateParams
  -> Handler
     -> ShouldBindQuery / ShouldBindJSON / FormFile
     -> 参数归一化
     -> 粗粒度业务校验
  -> Service
     -> 业务规则
     -> 状态流转
     -> repo 接口
  -> Repository
     -> GORM
     -> PostgreSQL
  -> OSS / Media
  -> response.Success / Fail*
```

结论：分层方向正确，但每条业务链路的完整程度不同。车辆链路最完整，公开端聚合链路最薄，参数模板链路存在真实断裂。

## 三、模块链路完整性

| 模块 | 当前链路 | 完整度 | 关键说明 |
|---|---|---:|---|

## 四、已确认缺陷

### P0：当前代码编译失败

位置：`internal/transport/http/helper/utils.go:53`

问题：`fmt.Errorf("字段%s不能为空！")` 中有 `%s` 占位符，但没有传入参数。`go test ./...` 编译到该包时失败。

影响：任何依赖该 helper 包的构建、测试、CI 都会失败；这是全链路交付的硬阻断。

企业解法：

- 编译错误必须进入 CI 的第一道门禁。
- 通用 helper 必须有最小单元测试。
- 错误函数不要只传值，建议传字段名：`RequiredField(fieldName string, value string)`。

### P1：参数模板 Update 链路 ID 断裂

位置：

- `handler/param_template.go:36`
- `handler/param_template.go:47`
- `repository/postgres/param_templates.go:28`
- `repository/postgres/param_templates.go:37`
- `repository/postgres/param_templates.go:47`

问题：路由参数 `:id` 被读取并传入 service，但 repository 的 `Update(ctx, p)` 实际依赖 `p.ID`。如果请求 body 不带 `id`，`p.ID` 可能为 nil，后续 `*p.ID` 会 panic 或更新不到目标记录。

影响：接口契约不稳定。前端以为 `PUT /:id` 的 id 是唯一来源，后端却隐式要求 body 也带 id。

企业解法：

- 路由 ID 是权威 ID，body 不允许覆盖。
- service 层应把 `id` 写入聚合对象，或 repo 方法签名改成 `Update(ctx, id string, patch ParamTemplatePatch)`。
- Update DTO 与 Entity 分离，避免直接用数据库模型接外部请求。

### P1：公开端链路不完整

位置：`internal/transport/http/router/router.go:100-104`

当前公开端只注册了：

```text
GET /api/v1/public/vehicles
```

缺失的 V1 常见公开端接口：

- `GET /api/v1/public/home`
- `GET /api/v1/public/categories`
- `GET /api/v1/public/vehicles/{id}`
- `GET /api/v1/public/contact`
- `GET /api/v1/public/vehicles/{id}/share-check`

影响：小程序无法完整消费系统设置、联系信息、分类导航、车辆详情、分享兜底等 V1 展厅核心能力。

企业解法：

- 公开端单独建 `public` service/handler，不复用管理端 handler 的路径判断。
- 用首页聚合接口减少小程序首屏请求数量。
- 公开端统一强制 `published` 过滤，避免草稿/下架/删除内容泄漏。

### P1：分类默认分页参数传递偏差

位置：`internal/transport/http/handler/categories.go:109`

问题：handler 已计算默认 `pageSize`，但构造 query 时传的是 `raw.PageSize`，不是归一化后的 `pageSize`。当请求不传 `pageSize` 时，service 看到的是 0，可能返回全量数据。

影响：接口行为与文档/代码意图不一致。数据量增大后会影响管理端性能。

企业解法：

- 参数归一化后只允许使用 normalized 值。
- 为分页边界写表驱动测试：未传、0、负数、超大值、正常值。
- 将分页解析沉淀为通用 `ParsePageQuery`，减少各 handler 重复实现。

### P1：发布校验仍是 MVP 放宽

位置：`handler/vehicles.go` 的 `Publish`

问题：发布车辆时只真实判断封面图，详情图和必填参数被写死为 true。

影响：不完整车辆可能进入公开展厅，破坏展示质量。

企业解法：

- 发布校验必须从数据库读取真实详情图、参数值、分类、价格策略。
- 状态机规则放在 domain/service，不放在 handler。
- 发布操作最好返回结构化缺失项，方便前端逐项提示。

### P1：媒体上传缺少补偿与生命周期管理

位置：`handler/upload.go`

问题：当前顺序是先上传 OSS，再写 `media_assets`。如果 DB 写入失败，OSS 对象会成为孤儿文件。

影响：存储成本增长，后台无法追踪文件归属，也无法安全删除。

企业解法：

- DB 失败时立即 best-effort 删除刚上传的 object。
- `media_assets` 增加 `status/ref_count/owner_type/owner_id`。
- 定时任务扫描未引用媒体并清理 OSS。
- 生产环境用异步图片压缩、缩略图、CDN 刷新链路。

### P2：批量状态更新不是事务

位置：`service/vehicle/service.go` 的 `BatchSetStatus`

问题：循环逐条读取并更新，中途失败会出现部分成功。

影响：管理端看到的批量操作结果可能不一致。

企业解法：

- 使用事务或单条 SQL 批量更新。
- 返回成功/失败明细。
- 对发布操作要逐条校验，批量下架可直接批量 SQL。

### P2：RBAC 能力已写，但未用于具体路由

位置：`middleware/jwtAuth.go`、`router/router.go`

问题：`RequireRole` 已实现，但 admin 路由只挂了 `JWTAuth`，没有按角色限制高危操作。

影响：所有登录用户都能访问所有管理接口。

企业解法：

- super_admin 才能删除、系统设置、用户管理。
- editor 可编辑车辆/分类，但不能改系统设置。
- viewer 只读。
- 权限策略集中声明，避免散落在 handler。

### P2：用户管理仍是 stub

位置：`router/router.go` 的 `/admin/user` 与 `handler/user.go`

问题：路由已暴露，但没有完整 service/repository/校验/权限模型。

影响：看起来有用户管理，实际不可作为生产能力。

企业解法：

- 要么从 V1 路由中隐藏，要么完整实现。
- 用户创建必须包含密码哈希、角色校验、禁用状态、审计日志。

## 五、企业级解决方案总览

### 1. 用“契约优先”管理前后端边界

企业项目不会只靠口头约定接口。推荐做法：

- OpenAPI 作为接口契约源头。
- 请求 DTO、响应 DTO、错误码、分页结构全部写清楚。
- 前端 SDK 从 OpenAPI 生成或至少由类型文件统一维护。
- 每次接口变更必须同步文档和契约测试。

在本项目中的落点：

- 补全 `api/openapi`。
- 明确 `public` 与 `admin` 两套接口契约。
- 不让 handler 直接返回 domain entity，改为 response DTO。

### 2. 用垂直切片检查每个功能是否闭环

不要只按目录看“有没有 handler/service/repo”，而要按业务动作看：

```text
入口路由
  -> 鉴权/权限
  -> 参数 DTO
  -> 参数校验
  -> service 业务规则
  -> repo 读写
  -> 事务/补偿
  -> 响应 DTO
  -> 日志/审计
  -> 测试
  -> 文档
```

以“参数模板更新”为例，当前有 handler/service/repo，但 ID 契约没有闭合，所以不能算完整。

### 3. 组合根继续保留，模块内部进一步收口

当前 `bootstrap.buildHandlers()` 的方向是对的，这是企业里常见的 composition root。

下一步建议：

- `router` 只收 handler，不认识 repo/DB。
- `handler` 只做协议转换，不做复杂业务。
- `service` 持有 repo 接口，承载业务规则和事务。
- `repository` 只处理持久化，不懂 HTTP。
- `domain/rule` 放状态机和纯规则。

### 4. 公开端与管理端分离

当前车辆 `List` 通过 URL 判断是否 public：

```go
onlyPublished := strings.Contains(c.Request.URL.Path, "/public/")
```

这能跑，但企业里更推荐显式分离：

```text
AdminVehicleHandler.List
PublicVehicleHandler.List
PublicVehicleHandler.Detail
PublicHomeHandler.Home
```

原因：

- 权限边界更清楚。
- 返回字段可以不同。
- public 永远强制 `published`。
- 管理端可以看 draft/unpublished/deleted。

### 5. 状态机与发布校验产品化

车辆状态建议明确成状态机：

```text
draft -> published
published -> unpublished
unpublished -> published
draft/unpublished/published -> deleted
deleted -> 不可编辑、不可发布、不可公开查询
```

发布前必须校验：

- 车辆名称。
- 分类存在且启用。
- 封面图存在。
- 详情图满足最低数量。
- 价格模式合法，显示价格时价格大于 0。
- 绑定分类的参数模板已启用。
- 必填参数均已填写。
- 免责声明/联系配置可用。

### 6. 媒体链路要有“补偿 + GC + CDN”

企业媒体链路通常不是简单上传：

```text
上传申请/直传
  -> OSS 临时对象
  -> 回调或 complete
  -> media_assets 落库
  -> 图片压缩/缩略图
  -> 引用到车辆/系统设置
  -> CDN 访问
  -> 无引用对象 GC
```

本项目当前是单步上传，适合本地开发和 MVP。进入生产前建议增加补偿删除和后台清理。

### 7. 质量门禁必须前置

当前 `go test ./...` 已发现编译失败，说明质量门禁还没形成闭环。

建议最小门禁：

```text
go fmt ./...
go vet ./...
go test ./...
go run ./cmd/migrate -op up
核心接口集成测试
OpenAPI 契约检查
```

推荐优先补的测试：

- `RequiredField` helper。
- 分类分页默认值。
- 参数模板 Update 不带 body.id。
- 车辆公开端只返回 published。
- 未登录访问 admin 返回 401。
- editor 访问系统设置返回 403。

## 六、推荐修复顺序

| 顺序 | 事项 | 原因 |
|---:|---|---|
| 1 | 修复 `RequiredField` 编译错误 | 当前构建硬阻断。 |

## 八、结论

当前后端的“工程骨架链路”已经搭起来了，方向是对的：配置、启动、组合根、路由、中间件、handler、service、repository、迁移、OSS 都有基本形态。

但从企业交付标准看，当前还不能判定为完整闭环，原因是：

- 编译存在 P0 错误。
- 参数模板 Update 存在真实 ID 契约断点。
- 公开端只完成车辆列表，展厅核心聚合接口缺失。
- 发布、媒体、RBAC、种子数据、测试门禁仍未闭环。

企业中的解法不是继续堆 handler，而是用“契约优先 + 垂直切片 + 状态机 + 事务/补偿 + CI 门禁 + 可观测性”把每条链路闭合。

## 九、补充：参数模板后端链路变化复盘

本次只按 `server` 目录内的未提交内容更新学习记录。重点不是管理端页面怎么消费，而是参数模板在 Go/Gin 后端里的链路如何从“只有创建/更新/详情”推进到“列表、删除、轻详情、完整详情都已接入”。

### 1. 本次后端变更范围

涉及文件：

- `server/internal/domain/model/param_templates.go`
- `server/internal/repository/postgres/param_templates.go`
- `server/internal/service/param_template/service.go`
- `server/internal/transport/http/handler/param_template.go`
- `server/internal/transport/http/handler/categories.go`
- `server/internal/transport/http/router/router.go`
- `server/internal/transport/http/constant/constant.go`

新增/调整的接口能力：

- `GET /api/v1/admin/param-templates/list`
- `GET /api/v1/admin/param-templates/getItemsbyId/:id`
- `GET /api/v1/admin/param-templates/getById/:id`
- `DELETE /api/v1/admin/param-templates/:id`

保留但仍需修复契约的接口：

- `PUT /api/v1/admin/param-templates/:id`

### 2. 领域模型变化

`model.ParamTemplate` 仍然是模板主表实体，新增了两个偏接口查询语义的类型：

```go
type PamListTmp struct {
    ParamTemplate
    ItemNum int `json:"itemNum"`
}

type TmpQuery struct {
    Page     int `form:"page"`
    PageSize int `form:"pageSize"`
}
```

学习点：

- `PamListTmp` 是列表响应 DTO，不是数据库实体本身。它通过 embedding 复用 `ParamTemplate` 字段，再额外暴露 `itemNum`。
- `TmpQuery` 是 query DTO，依赖 `form` tag 被 `ShouldBindQuery` 读取。
- 命名上仍有改进空间：`PamListTmp`/`TmpQuery` 语义偏弱，后续建议收敛为 `ParamTemplateListItem` / `ParamTemplateListQuery`。

### 3. Repository 变化

新增能力：

```text
List(ctx, q)
GetById(ctx, id)
GetItemsById(ctx, id)
Delete(ctx, id)
ItemsCount(ctx, id)
```

关键点：

- `List` 使用 `Offset/Limit` 做分页，排序为 `created_at DESC`。
- `GetById` 只查模板主表，不加载 `Items`。
- `GetItemsById` 使用 `Preload("Items")` 加载一对多子项。
- `ItemsCount` 针对每个模板统计 `param_template_items` 数量，用于列表页轻量展示。
- `Delete` 删除主表记录，依赖外键 `ON DELETE CASCADE` 清理子项。

易踩坑：

- `ItemsCount` 当前由 service 层对列表逐条并发调用，属于 N+1 查询的并发版。数据量小时能跑，数据量变大后应考虑用 `LEFT JOIN + GROUP BY` 一次查出 `itemNum`。
- `Delete` 注释里提到 `category_id ... ON DELETE RESTRICT`，但删除模板时真正需要关注的是 `param_template_items.template_id ON DELETE CASCADE`。注释建议修正，避免后来者误解。

### 4. Service 变化

`ParamTemplateService.List` 当前流程：

```text
Count
  -> List
  -> 为每条模板并发 ItemsCount
  -> 组装 response.ListResult[model.PamListTmp]
```

学习点：

- `sync.WaitGroup` 类似等待一批并发任务全部完成；goroutine 中显式传入 `index` 和 `id`，避免闭包捕获循环变量。
- 这里忽略了 `ItemsCount` 的错误：`ic, _ := ...`。企业代码里不建议吞错，至少要收集第一个错误并返回，或者降级为 `itemNum=0` 同时记录日志。
- `count == 0` 时当前直接返回 `response.ListResult[model.PamListTmp]{}`，这会导致 `list/page` 都是零值。更稳的契约是返回：

```json
{
  "list": [],
  "page": {
    "page": 1,
    "pageSize": 10,
    "total": 0,
    "totalPages": 0
  }
}
```

- `TotalPages` 当前用 `math.Ceil(float64(c) / float64(q.PageSize))`，前提是 `q.PageSize > 0`。虽然 handler 已兜底默认值，但 service 作为业务层仍应避免隐式依赖，或者把分页归一化做成公共函数。

### 5. Handler 与 Router 变化

参数模板路由从原来的：

```text
POST   /admin/param-templates
PUT    /admin/param-templates/:id
GET    /admin/param-templates/all-detail/:id
GET    /admin/param-templates/detail/:id
```

调整为：

```text
GET    /admin/param-templates/list
POST   /admin/param-templates
PUT    /admin/param-templates/:id
GET    /admin/param-templates/getItemsbyId/:id
GET    /admin/param-templates/getById/:id
DELETE /admin/param-templates/:id
```

学习点：

- `middleware.ValidateParams([]string{"page", "pageSize"})` 被挂到列表路由，说明 query 白名单校验已经进入参数模板链路。
- `GetById` 与 `GetItemsById` 分离是合理方向：前者轻量查主表，后者查完整模板和子项。
- 路由命名建议后续统一 REST 风格或统一动词大小写。当前 `getItemsbyId` 中 `by` 小写不一致，长期会增加记忆成本。

### 6. 常量抽取

新增 `server/internal/transport/http/constant/constant.go`，把分类分页常量从 handler 内部抽出：

```go
const (
    DEFAULT_CATEGORY_LIST_PAGE      = 1
    DEFAULT_CATEGORY_LIST_PAGE_SIZE = 10
    MAX_CATEGORY_LIST_PAGE_SIZE     = 100
)
```

当前参数模板列表也复用了这组常量。学习点：

- 抽公共常量能减少重复，但命名不应绑定单一业务对象。既然参数模板也使用，建议后续改名为 `DefaultListPage`、`DefaultListPageSize`、`MaxListPageSize`。
- 常量包位于 `transport/http` 下，说明它服务 HTTP 请求解析层，不应被 repository 或 domain 层依赖。

### 7. 仍未闭环的问题

#### Update 仍未只信任 URL id

当前 `Update(ctx, id, q)` 先用 URL id 查存在性，但 repo 更新仍依赖 `q.ID`：

```text
handler 读取 :id
  -> service GetById(id) 确认存在
  -> repo.Update(q) 依赖 body 中的 q.ID
```

正确方向：

```text
handler 读取 :id
  -> body 只承载可修改字段
  -> service/repo 使用 URL id 作为唯一主键
```

建议下一步改成：

```go
Update(ctx context.Context, id string, p *model.ParamTemplate)
```

并在 service 层把 `id` 写回聚合根，或在 repo 层用 `Where("id = ?", id)` 明确更新目标。

#### 列表接口空结果响应不稳定

`count == 0` 时应仍返回稳定的 `{ list, page }`。这类契约细节不只是前端体验问题，也是接口可测试性问题。

#### `ItemsCount` 不应吞错

并发查询中忽略错误会让接口看起来成功，但数据不可信。建议至少使用 channel 收集错误：

```text
goroutine ItemsCount
  -> error channel
  -> Wait 后统一判断
```

或者先用简单串行实现，等确实有性能压力再优化。

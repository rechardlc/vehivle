## 一、前端 vs 后端 概念对照

| 前端概念 | 后端对应 | 说明 |
|----------|----------|------|
| `npm run dev` | `go run ./cmd/api` / `make run` | 启动命令，Go 无官方脚手架 |
| `package.json` scripts | Makefile | 任务编排与常用命令封装 |
| `import` 模块 | `import` 包 | 按包路径导入，不是按文件 |
| `node_modules` | `go.mod` + `go.sum` + 模块缓存 | 依赖管理 |
| 路由 (React Router) | Gin Router | 路径与 Handler 映射 |
| 中间件 (Express/Koa) | Gin Middleware | 请求前后处理链 |
| API 请求 (fetch/axios) | Handler 接收请求、返回 JSON | 服务端处理 HTTP |
| 环境变量 (.env) | 环境变量 + configs | 配置管理 |
| 事务 | 数据库事务 | 多步操作原子性 |
| 鉴权 (JWT) | JWT 签发/解析、RBAC | 身份与权限 |
| 迁移 (Prisma migrate) | SQL 迁移脚本 | 数据库结构版本管理 |
| 幂等 | 重复请求结果一致 | 防重复提交、重试安全 |

## 二、Go 模块基础（已掌握）

### 2.1 初始化模块

```bash
go mod init <模块名>
# 例如：go mod init vehivle
```

- **作用**：创建 `go.mod`，定义模块路径（所有包导入的前缀）
- **模块名**：可任意，常用 `github.com/用户名/仓库名` 或简短名如 `vehivle`

### 2.2 直接依赖 vs 间接依赖

| 类型 | 含义 | 在 go.mod 中 |
|------|------|--------------|
| **直接依赖** | 你的代码 `import` 的包 | 无 `// indirect` |
| **间接依赖** | 直接依赖所依赖的包 | 有 `// indirect` |

- 企业实践：直接依赖显式声明，`go get <包>` 可将其标为直接依赖
- `go mod tidy`：清理未用依赖，整理 indirect 标记

### 2.3 导入解析

- `go run ./cmd/api` 从**模块根目录**（`go.mod` 所在）解析所有 import
- 导入的是**包路径**，不是文件路径：`import "vehivle/internal/bootstrap"`
- 同一目录下所有 `.go` 文件属于同一包，Go 自动一起编译

## 三、项目结构与启动（已掌握）

### 3.1 目录结构说明

```
server/
├── go.mod              # 模块定义
├── go.sum              # 依赖校验和
├── cmd/api/main.go     # 入口
├── configs/            # 配置（YAML + 环境变量，viper 加载）✅
│   ├── conf.go         # 配置结构、Load、校验
│   ├── default.go      # 默认值常量
│   ├── conf.dev.yaml   # 开发环境配置
│   └── conf.prod.yaml  # 生产环境配置
├── internal/           # 内部包，不可被外部 import
│   └── bootstrap/      # 依赖注入（待落地）
├── pkg/                # 可复用公共包
│   ├── logger/         # 日志（slog、RequestID/AccessLog 中间件）✅
│   └── response/       # 统一响应 ✅
└── ...
```

> 当前各模块完成状态见 [progress.md](./progress.md)。

### 3.2 启动方式

```bash
cd server
go run ./cmd/api
# 或
make run   # 若已配置 Makefile
```

**注意**：工作目录需为 `server/`，以便加载 `.env`、`configs/conf.dev.yaml` 等文件。

### 3.3 main 启动流程（当前实现）

1. `configs.Load()` 加载配置，错误则 `os.Exit(1)`
2. 按 `cfg.App.Env` 设置 `gin.SetMode`（prod → ReleaseMode）
3. `gin.New()` 创建引擎，注册中间件：`gin.Logger`、`gin.Recovery`、`logger.RequestID`、`logger.AccessLog`
4. 注册路由（如 `GET /health`），使用 `response.Success` 返回
5. `r.Run(":9991")` 启动服务

### 3.4 Makefile 基础

- **Makefile**：定义任务（target）与对应命令
- **make**：读取 Makefile 并执行，如 `make run`、`make build`
- 作用类似 `package.json` 的 scripts，企业项目常用

## 四、Go 类型转换（logger.Env）

- **问题**：`cfg.App.Env` 是 `string`，`NewLogger(env Env)` 需要 `Env` 类型
- **原因**：Go 强类型，`string` 与 `type Env string` 是不同类型
- **解决**：显式转换 `logger.Env(env)`
- **好处**：类型安全，只接受合法环境值

## 五、Gin 基础（已掌握）

### 4.1 最小示例

```go
r := gin.Default()                    // 默认引擎（含 Logger + Recovery）
r.GET("/health", func(c *gin.Context) {
    c.JSON(http.StatusOK, gin.H{      // gin.H = map[string]interface{}
        "status":  "ok",
        "message": "API is running",
    })
})
r.Run(":9991")                        // 监听端口，阻塞运行
```

### 4.2 关键概念

| 概念 | 说明 |
|------|------|
| `r` | `*gin.Engine` 类型，Gin 核心引擎，负责路由、中间件、请求处理 |
| `gin.Default()` | 默认引擎，内置 Logger、Recovery 中间件 |
| `gin.New()` | 空引擎，无中间件，企业开发常用，便于完全自定义 |
| `gin.H` | `map[string]interface{}` 简写，用于构建 JSON |
| `c *gin.Context` | 请求上下文，每次请求新建，包含请求/响应/参数等 |
| `c.JSON()` | 返回 JSON 响应 |
| `r.Run()` | 启动 HTTP 服务，阻塞直到退出 |

### 4.3 gin.Default() vs gin.New()

| 方法 | 说明 | 适用场景 |
|------|------|----------|
| `gin.Default()` | 已带 Logger + Recovery | 开发、快速原型 |
| `gin.New()` | 空引擎，无中间件 | 企业生产，完全自定义 |

企业自定义示例：

```go
r := gin.New()
r.Use(gin.Logger())
r.Use(gin.Recovery())
r.Use(customLogger)      // 接入公司日志规范
r.Use(customRecovery)    // panic 上报 + 统一错误响应
// 业务路由...
```

## 六、Go 指针与 import 可见性（已掌握）

### 5.1 指针基础

| 符号 | 含义 | 示例 |
|------|------|------|
| `*T` | 指向类型 T 的指针类型 | `*gin.Engine`、`*gin.Context` |
| `&v` | 取变量 v 的地址 | `p := &x` |
| `*p` | 解引用，访问 p 指向的值 | `*p = 100` |

**作用**：在函数间共享/修改同一份数据、避免大结构体拷贝、表示“可能为空”。

**注意**：`*` 和 `&` 不必同时出现，按需使用；空指针解引用会 panic。

### 5.2 企业开发中的使用频率

- **多用指针**：结构体、服务、仓储、配置、框架类型（如 `*gin.Context`）
- **少用指针**：基本类型（int、string）、小结构体、只读数据

### 5.3 import 与可见性

- **使用方式**：`import` 后需加包名前缀，如 `gin.Default()`、`http.StatusOK`
- **可见性规则**：首字母**大写** = 导出（包外可用），首字母**小写** = 未导出（包外不可见）
- **前端类比**：类似 `import * as gin from 'gin'`，导出规则类似 JS 的 `export`

### 5.4 指针深入（前端开发者视角）

#### 5.4.1 类型 vs 值

| 概念 | 含义 | 例子 |
|------|------|------|
| **类型 (Type)** | 数据的「模板」 | `defaultLogger` 是结构体类型 |
| **值 (Value)** | 按类型创建出的「具体数据」 | `defaultLogger{...}` 是结构体值（实例） |

在 Go 中，「值」是相对于「指针」而言的：值 = 数据本身，指针 = 指向数据的地址。结构体实例就是一种值。

#### 5.4.2 何时用指针 vs 值

| 场景 | 前端类比 | Go 建议 |
|------|----------|---------|
| 需要修改原数据 | 修改对象属性 | 用指针 `*T` |
| 数据较大 | 大对象/大配置 | 用指针 `*T` |
| 可能为空 | `null` / `undefined` | 用指针 `*T` |
| 小数据、只读 | 数字、字符串 | 用值 `T` |
| 库返回指针 | 库返回引用 | 跟着用指针 |

#### 5.4.3 接收者是指针时，`l` 已是指针

```go
func (l *defaultLogger) Debug(...) Logger {
    return l   // l 本身就是 *defaultLogger，无需 &l
}
```

- 接收者 `(l *defaultLogger)` 表示 `l` 的类型是 `*defaultLogger`（指针）
- `return l` 就是在返回指针，不需要 `&l`（`&l` 会得到 `**defaultLogger`）
- `&` 用于对「值」取地址；当变量已是指针时，直接使用即可

#### 5.4.4 With 方法的数据流

```go
func (l *defaultLogger) With(args ...any) Logger {
    log := l.Logger.With(args...)   // ① 得到 *slog.Logger（指针）
    return &defaultLogger{log}      // ② ③
}
```

| 步骤 | 得到 | 说明 |
|------|------|------|
| ① | `*slog.Logger` | slog 的 With 返回指针 |
| ② | `defaultLogger{log}` | 结构体值，内部字段存的是指针 |
| ③ | `&defaultLogger{log}` | 对结构体值取地址，得到 `*defaultLogger` |

指针作为字段放入结构体 → 得到结构体值 → 再对结构体取地址得到 `*defaultLogger`。

```go
cfg, err := configs.Load()
if err != nil {
    slog.Error("failed to load config", "error", err)

// 错误检查必须先于使用 cfg
```

- **debug 模式**：开发环境，输出路由等调试信息

```go
switch cfg.App.Env {
case "prod":
    gin.SetMode(gin.ReleaseMode)

    gin.SetMode(gin.DebugMode)
}
```

## 八、pkg/logger 日志（已掌握）

### 7.1 核心概念

- **Logger 接口**：定义 Info、Error、Warn、Debug、Fatal、With 方法，便于 mock 与替换实现
- **defaultLogger**：基于 `log/slog` 的实现，组合 `*slog.Logger`
- **环境区分**：`EnvDev` / `EnvProd`，生产环境用 Info 级别、UTC 时间、JSON 格式
- **Env 类型**：`type Env string`，`NewLogger` 接收 `Env` 而非 `string`，需显式转换

### 7.2 使用方式

```go
// 创建 Logger（configs 返回 string，需转换为 logger.Env）
env := cfg.App.Env  // string
log := logger.NewLogger(logger.Env(env))

// 记录日志（支持 context）
log.Info(ctx, "message", "key", "value")
log.Error(ctx, "error occurred", "err", err)

// 添加上下文后返回新 Logger
log.With("request_id", "xxx").Info(ctx, "request processed")
```

### 7.3 中间件（middleware.go）

| 中间件 | 作用 |
|--------|------|
| `RequestID()` | 从 `X-Request-ID` 读取或生成 UUID，写入 `c.Set(GinRequestIDKey, requestID)`，响应头回传 |
| `AccessLog(logger)` | 记录请求 path、method、client_ip、request_id、status、latency |

```go
r.Use(logger.RequestID())
r.Use(logger.AccessLog(logger.NewLogger(logger.Env(env))))
```

- **GinRequestIDKey**：`"request_id"`，与 `pkg/response` 的 `getRequestID(c)` 共用同一 key

### 7.4 与 slog 的对应关系

| 方法 | slog 对应 |
|------|-----------|
| `Info` | `InfoContext` |
| `Error` | `ErrorContext` |
| `With` | `With`（返回带上下文的 Logger） |

### 7.5 内部实现要点

- **嵌入 `*slog.Logger`**：`slog.New()` 和 `slog.Logger.With()` 都返回 `*slog.Logger`，嵌入指针与 slog API 一致，避免拷贝。
- **返回类型 `Logger`**：方法返回 `Logger` 接口，实际返回的是 `*defaultLogger` 指针；接口可以持有指针，调用方得到的是接口类型。
- **With 链式调用**：`log.With("key", "val")` 返回新的 `*defaultLogger`，内部包含 slog 的带上下文 Logger，实现链式添加上下文。

## 九、pkg/response 统一响应（已掌握）

### 8.1 结构体与字段

```go
type Body struct {
    Code      string      `json:"code"`
    Message   string      `json:"message"`
    Data      interface{} `json:"data"`   // 任意类型，类似 JS 的 any
    RequestID string      `json:"request_id"`
    Timestamp string      `json:"timestamp"`
}
```

- **字段大写**：导出字段才能被 `encoding/json` 序列化
- **`interface{}`**：表示任意类型，各接口返回的 data 结构不同，统一用此类型

### 8.2 错误码常量

| 常量 | 值 | 说明 |
|------|-----|------|
| `CodeSuccess` | 000000 | 成功 |
| `CodeAuthFailed` | A00001 | 认证失败 |
| `CodeAuthDenied` | A00004 | 授权失败 |
| `CodeParamError` | B00001 | 参数错误 |
| `CodeBusinessError` | C00002 | 业务错误 |
| `CodeMediaError` | M00001 | 媒体错误 |

### 8.3 使用方式

```go
// 成功响应
response.Success(c, gin.H{"id": 1, "name": "xxx"})
response.Success(c, nil)  // data 为 nil 时自动返回 {}

// 失败响应（Fail 签名为 c, code, message）
response.Fail(c, response.CodeBusinessError, "当前状态不允许操作")
response.FailParam(c, "参数不能为空")
response.FailAuth(c, "token 无效")
response.FailAuthDenied(c, "无权限")
response.FailBusiness(c, "车型已下架")
```

### 8.4 request_id 与中间件配合

- `logger.RequestID()` 将 request_id 写入 `c.Set("request_id", ...)`
- `response.getRequestID(c)` 从 `c.Get("request_id")` 读取，填充到 `Body.RequestID`
- 需保证 RequestID 中间件在 AccessLog、业务 Handler 之前注册

### 8.5 前端 vs 后端 对照

| 前端 | 后端 |
|------|------|
| `res.data` 取业务数据 | `Body.Data` 对应 `data` 字段 |
| 根据 `code` 判断成功/失败 | `response.Success` / `response.Fail` 统一设置 |
| 错误提示用 `message` | `Body.Message` 对应 |

## 十一、Go 泛型扩展（可选了解）

### 10.1 基本语法

```go
// 函数泛型：类型参数放在函数名后
func Id[T any](x T) T { return x }

// 结构体泛型
type Box[T any] struct { Value T }
```

### 10.2 常用约束

| 约束 | 含义 |
|------|------|
| `any` | 任意类型（等同 `interface{}`） |
| `comparable` | 可比较（支持 `==`、`!=`） |
| `~int` | 底层为 int 的类型（含 `type MyInt int`） |

### 10.3 示例

```go
// 泛型切片操作
func Map[T, U any](s []T, f func(T) U) []U {
    out := make([]U, len(s))
    for i, v := range s { out[i] = f(v) }
    return out
}

// 泛型查找
func Contains[T comparable](s []T, v T) bool {
    for _, x := range s { if x == v { return true } }
    return false
}
```

### 10.4 与 interface{} 的选择

- **JSON 响应 Data 字段**：用 `interface{}` / `any` 更合适，各接口返回类型不同
- **容器、工具函数**：用泛型获得类型安全与代码复用

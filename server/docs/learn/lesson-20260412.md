## 一、为什么要做这次重构

重构前，每个 handler 的构造函数都在「自己内部」做三件事：

1. 拿着 `*gorm.DB` 去 `new` 具体的 postgres repo
2. 用 repo 去 `new` service
3. 把 service 存进自己的结构体

问题在于：

- **Handler 知道了太多**：它 import 了 `postgres` 包、知道 repo 怎么建，但 handler 的本职是「HTTP 请求进来 → 调 service → 返回 JSON」
- **无法替换实现**：想在测试里用假 repo？做不到，因为组装写死在 handler 构造函数里
- **Router 也被拖下水**：router 必须拿 `*gorm.DB` 才能传给各个 `handler.NewXxx(db, ...)`

前端类比：

- **重构前**像在 React 页面组件里 `new HttpClient()` + `new Repository(client)` + `new Store(repo)`——页面耦合了所有底层实现
- **重构后**像在 `App.tsx` / `providers.tsx` 里统一初始化，页面只收 `useStore()` 返回的接口

## 二、组合根（Composition Root）是什么

**组合根**是应用里「唯一知道所有具体实现」的地方，在本项目就是 `bootstrap.go`。

```
main.go
  └─ bootstrap.New(cfg)
       └─ Run()
            ├─ pgsqlConnPool()       → b.db
            ├─ ossConnPool()         → b.ossClient
            ├─ buildHandlers()       → 组装 repo → service → handler  ← 这就是组合根
            └─ injectRouter(engine)  → 把装好的 handlers 交给 router
```

**规则很简单**：

| 层 | 可以 import | 不可以 import |
|-----|-----------|-------------|
| bootstrap（组合根） | 所有包 | — |
| router | handler、middleware | repository、service |
| handler | service（接口/结构体） | repository |
| service | domain（model/enum/rule）、repo 接口 | 具体 postgres 实现 |
| repository | domain、gorm | — |

## 四、每个 handler 改了什么

### 4.1 Auth

| | 重构前 | 重构后 |
|--|--------|--------|
| 构造函数 | `NewAuth(db, jwtCfg)` | `NewAuth(authSvc, jwtCfg)` |
| 内部做 | `postgres.NewUserRepo(db)` → `auth.NewService(repo, ...)` | 直接收 `*auth.Service` |
| import postgres | 是 | 否 |
| import gorm | 是 | 否 |

### 4.2 User（stub）

| | 重构前 | 重构后 |
|--|--------|--------|
| 结构体 | `User { DB *gorm.DB }` | `User {}` |
| 构造函数 | `NewUser(db)` | `NewUser()` |

User 模块尚无 service/repo，方法体都是占位返回。移除 DB 字段让它更诚实——没用到就不挂。

### 4.3 Categories

| | 重构前 | 重构后 |
|--|--------|--------|
| 结构体 | `Categories { DB, CategoryService }` | `Categories { CategoryService }` |
| 构造函数 | `NewCategories(db)` | `NewCategories(catSvc)` |
| 内部做 | `postgres.NewCategoryRepo(db)` → `NewCategoryService(repo)` | 直接收 `*category.CategoryService` |

### 4.4 Vehicles

| | 重构前 | 重构后 |
|--|--------|--------|
| 构造函数 | `NewVehicles(db, oss)` | `NewVehicles(vehSvc, catSvc, mediaRepo, oss)` |
| 内部做 | new 三个 repo + 两个 service | 直接收已装配好的 service 和 repo |

### 4.5 System

| | 重构前 | 重构后 |
|--|--------|--------|
| 结构体 | `System { DB, OSS, SysService }` → 之前已去掉 DB | `System { OSS, SysService }` |
| 构造函数 | `NewSysSettings(db, oss)` | `NewSysSettings(sysSvc, oss)` |

### 4.6 Upload

| | 重构前 | 重构后 |
|--|--------|--------|
| 结构体 | `Upload { OSS, DB }` | `Upload { OSS, mediaRepo }` |
| 落库方式 | `u.DB.WithContext(ctx).Create(row).Error` | `u.mediaRepo.Create(ctx, row)` |

Upload 原来直接拿 `*gorm.DB` 写 SQL，现在改为调 `MediaAssetRepo.Create()`，SQL 细节收进 repo。

## 五、Router 的变化

### 5.1 新增 `Handlers` 结构体

```go
type Handlers struct {
    Auth       *handler.Auth
    User       *handler.User
    Vehicles   *handler.Vehicles
    Categories *handler.Categories
    System     *handler.System
    Upload     *handler.Upload
}
```

Router 的构造函数从 `New(engine, logger, db, oss, jwt)` 变成 `New(engine, logger, oss, jwt, handlers)`——**不再需要 `db`**。

### 5.2 路由注册

```go
// 重构前：router 自己 new handler
sysHandler := handler.NewSysSettings(r.db, r.oss)
sys.GET("", sysHandler.Detail)

// 重构后：直接用传进来的 handler
sys.GET("", r.handlers.System.Detail)
```

Router 回归了本职——**只管路由注册和中间件挂载**，不碰任何业务对象的创建。

## 六、Bootstrap `buildHandlers()` 详解

```go
func (b *Bootstrap) buildHandlers() *router.Handlers {
    // 1. Repos：全部用 b.db 创建
    userRepo  := pgrepo.NewUserRepo(b.db)
    catRepo   := pgrepo.NewCategoryRepo(b.db)
    vehRepo   := pgrepo.NewVehicleRepo(b.db)
    sysRepo   := pgrepo.NewSysSettings(b.db)
    mediaRepo := pgrepo.NewMediaAssetRepo(b.db)

    // 2. Services：收 repo 接口
    authSvc := auth.NewService(userRepo, auth.JWTConfig{...})
    catSvc  := category.NewCategoryService(catRepo)
    vehSvc  := vehicle.NewService(vehRepo)
    sysSvc  := system_setting.NewSysService(sysRepo)

    // 3. Handlers：收 service，不碰 db
    return &router.Handlers{
        Auth:       handler.NewAuth(authSvc, b.cfg.JWT),
        User:       handler.NewUser(),
        Vehicles:   handler.NewVehicles(vehSvc, catSvc, mediaRepo, b.ossClient),
        Categories: handler.NewCategories(catSvc),
        System:     handler.NewSysSettings(sysSvc, b.ossClient),
        Upload:     handler.NewUpload(b.ossClient, mediaRepo),
    }
}
```

**阅读顺序**：从上到下，正好就是依赖关系：repo → service → handler。这就是「组合根」的价值——**一个函数看完全应用的装配**。

前端类比：这像 Next.js `app/providers.tsx` 里统一创建 QueryClient、AuthProvider、StoreProvider，然后嵌套传下去。页面组件（handler）只调 `useQuery()`（service），不管底层是 fetch 还是 axios（repo）。

## 七、前端 vs Go 依赖注入对照

| 概念 | 前端（React） | Go 后端 |
|------|--------------|---------|
| 组合根 | `App.tsx` / `providers.tsx` | `bootstrap.buildHandlers()` |
| 注入方式 | Context / Props / DI 容器 | 构造函数参数 |
| 接口 | TypeScript `interface` | Go `interface`（隐式实现） |
| 具体实现 | `HttpClient`、`LocalStorageAdapter` | `postgres.VehicleRepo`、`oss.MinioClient` |
| 页面/处理器 | `PageComponent({ api })` | `handler.NewVehicles(svc, ...)` |
| 测试替换 | `jest.mock()` / 注入假 client | `NewService(fakeRepo)` |

Go 没有 `@Injectable()` 装饰器，也没有 Angular 那样的 DI 容器。大多数 Go 项目用**最朴素的方式**——构造函数传参。简单、显式、零魔法。

企业里如果项目大了，会用 **Google Wire**（编译时代码生成）或 **uber/fx**（运行时反射注入）来自动化 `buildHandlers()` 这类样板代码。但手写构造函数注入在中小项目完全够用。

## 八、这次学到的关键点

### 8.1 Handler 不应该知道「数据库怎么连」

Handler 的职责是：解析 HTTP 请求 → 调用 service → 组装 HTTP 响应。它不需要知道 service 底下是 Postgres 还是 MongoDB。

### 8.2 Router 不应该做「对象创建」

Router 的职责是：把 URL + Method 映射到 handler 函数。它不需要知道 handler 依赖了哪些 service。

### 8.3 `*gorm.DB` 只在两个地方出现

1. `bootstrap`：创建连接
2. `repository/postgres`：执行查询

中间所有层（service、handler、router）都不碰它。这就是分层的目标。

### 8.4 结构体字段是诚实的

重构前 `Categories { DB *gorm.DB, CategoryService }` 里的 `DB` 从没被方法用过，属于"挂了但没用"。重构后去掉了，结构体只留真正需要的依赖。
